# Phase 02.3: Form Input Validation - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Add client-side validation UX to all existing manage forms (ListingForm, BrandDialog, CategoryDialog, SkuInlineEditor). This includes validation timing, error display, required field indicators, and restructuring the API error format for future field-level mapping. No new forms or features — strictly improving validation on what exists.

</domain>

<decisions>
## Implementation Decisions

### Validation Timing
- Validate fields on blur (when user leaves the field)
- After an error appears, clear it on change as soon as the input becomes valid (instant positive feedback)
- Also validate all fields on submit as a final catch-all (catches untouched required fields)
- On submit with errors, scroll to and focus the first invalid field (react-hook-form `shouldFocusError`)

### Server Error Handling
- Keep toast-only display of server validation errors for now (field-level mapping deferred to future phase)
- Restructure API error response format NOW to support future field mapping: `{ error: 'Validation failed', fieldErrors: { name: 'message' }, formErrors: ['message'] }` — follows Zod's `flatten()` output shape
- HTTP status code only for categorization (no status in response body)
- Update `apiFetch` client utility to parse the new structured error format
- Create custom `ApiError` class extending `Error` with `fieldErrors`, `formErrors`, `status` properties
- Non-validation errors (500s, network failures) stay as toast — no special treatment needed
- Disable submit button with loading state during API call (react-hook-form `isSubmitting`)

### SKU Row Validation
- Validate on blur per field — consistent with main form timing
- Red border on invalid fields + tooltip on hover showing error message (compact for table rows)
- Reference shadcn React Hook Form recommended implementation for error display patterns
- Price and MSRP fields: validate decimal pattern on blur only (no input keystroke restriction)
- Row-level error highlight: defer to shadcn patterns (Claude's discretion)

### Required Field Indicators
- Red asterisk on required field labels (standard `*` indicator)
- Character counter shown only on textarea fields (e.g., description 142/5000) — not on short inputs
- No "(optional)" labels — asterisks on required fields are sufficient
- Select/dropdown fields (brand, status, categories) get the same treatment: required asterisk + error on blur if empty

### Claude's Discretion
- Exact implementation of tooltip error display on SKU rows
- Row-level error highlight styling (defer to shadcn patterns)
- Whether to install shadcn Form components or keep raw react-hook-form register/errors pattern
- Loading spinner vs text change on submit button
- Exact character counter styling and positioning

</decisions>

<specifics>
## Specific Ideas

- User referenced shadcn React Hook Form documentation (https://ui.shadcn.com/docs/forms/react-hook-form) as the implementation reference for form validation patterns
- API error format restructuring is a prep step — display stays toast-only now, but the structured format enables field mapping in a future phase
- ApiError class should be instanceof-checkable so mutation hooks can differentiate validation errors from other API errors

</specifics>
