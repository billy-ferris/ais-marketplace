# Phase 02.3 Plan 03: SKU Inline Editor Validation Summary

Retrofitted the non-RHF `SkuInlineEditor` with a local, blur-triggered validation layer that reuses the shared `createSkuSchema` (red border + Base UI Tooltip on invalid cells) without converting it to `useFieldArray`, and wired a submit-time catch-all so `ListingForm` blocks the save when any non-deleted SKU row is invalid or untouched-empty.

## What Was Built

**Task 1 — Per-cell validation state + validateAllRows** (`29c9432`)
- Imported `createSkuSchema` from `@ais/shared`.
- Added `skuErrors` state keyed by `` `${index}:${field}` `` (`''` = valid).
- Added `buildSkuCandidate(row)` helper coercing the row's string values into the schema's expected input shape (optional string/number fields collapse empty strings to `undefined`; `quantity` → `Number(...) || 0`).
- `validateSkuField(index, field, row)` runs `createSkuSchema.safeParse` and records the first issue whose `path[0] === field` (or clears it).
- Wired `onBlur` on the four required cells (name, price, msrp, quantity); price/MSRP get NO onChange numeric filtering (blur-only per CONTEXT). Each required cell's onChange re-runs the validator only when that cell currently has an error (clear-on-valid, mirroring `reValidateMode: 'onChange'`).
- Added `validateAllRows(): boolean` iterating every non-deleted row, writing errors for the four required fields and returning true only when all non-deleted rows fully pass. Deleted (`_deleted`) rows are excluded from all validation.

**Task 2 — Red border + Base UI Tooltip under a mounted provider** (`42a32b2`)
- Imported `Tooltip`, `TooltipTrigger`, `TooltipContent`, `TooltipProvider`.
- Mounted `<TooltipProvider>` once wrapping the editor's returned tree (RESEARCH Pitfall 5 — provider was exported but never mounted).
- Added `renderRequiredCell(errorKey, input)` helper: wraps a required-cell `Input` in `<Tooltip><TooltipTrigger render={input} /><TooltipContent>{message}</TooltipContent></Tooltip>` when the cell has an error, else returns the Input unchanged.
- Set `aria-invalid={!!skuErrors[key]}` on each required Input so the existing `aria-invalid:border-destructive` styling produces the red border (no explicit border class needed).
- Optional cells (sku, upc, size, casePack, casesPerPallet, image) and the `_deleted`/Add-SKU rendering are unchanged.

**Task 3 — validateAll via ref + ListingForm submit gate** (`6e6b092`)
- Converted `SkuInlineEditor` to `forwardRef<SkuInlineEditorHandle, SkuInlineEditorProps>` and added `useImperativeHandle(ref, () => ({ validateAll: validateAllRows }), [validateAllRows])`.
- Exported the `SkuInlineEditorHandle` type (`{ validateAll: () => boolean }`).
- In `ListingForm`: added `skuEditorRef = useRef<SkuInlineEditorHandle>(null)`, passed `ref={skuEditorRef}` to `<SkuInlineEditor />`, and in `onFormSubmit` added `const skusValid = skuEditorRef.current?.validateAll() ?? true; if (!skusValid) return;` BEFORE building/sending the payload.
- 02.3-02's ListingForm changes (mode config, RequiredLabel, FieldError, CharCounter, brand trigger, status handling, create-vs-edit branching) left intact.

## Decisions Made

- **No `useFieldArray` migration.** SkuInlineEditor keeps its existing array + `_deleted`/undo/image-upload semantics; validation is a local `skuErrors` layer over `createSkuSchema.safeParse` (RESEARCH anti-pattern honored).
- **`validateAllRows` fails on any row parse failure.** It returns false whenever a non-deleted row fails `safeParse` (even for an optional-field issue), matching "all non-deleted rows pass," but only surfaces error messages on the four required fields.
- **`aria-invalid` drives the red border.** The shared Input already applies `aria-invalid:border-destructive`, so setting `aria-invalid` was sufficient — no redundant `border-destructive` class added.
