# Phase 02.3: Code Review Report

**Reviewed:** 2026-07-01T02:19:07Z
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

Reviewed the full validation-error-contract plumbing (`error.ts`, `api.ts`, the three mutation-hook files) and the form-timing/indicator work (`FormFieldHelpers`, `BrandDialog`, `CategoryDialog`, `ListingForm`, `SkuInlineEditor`). The plumbing layer (Plan 01) is solid and well-tested. The UI layer (Plans 02/03) has two shipped-but-broken behaviors that directly contradict what the UI visually promises the user:

1. `SkuInlineEditor`'s submit-time catch-all (`validateAllRows`) can silently block the Save button with **zero visible error** when the invalid field is `casePack`, `casesPerPallet`, `sku`, `upc`, or `size` — only `name`/`price`/`msrp`/`quantity` get a tooltip/red-border. A user typing `0` into "Case Pack" (a very natural value to type) gets a Save button that does nothing.
2. `BrandDialog` marks "Manufacturer" as required (red asterisk, blur-triggered `trigger('companyId')`, inline `FieldError`) but `createBrandSchema.companyId` is `.optional()`, so the field is never actually enforced — an admin can submit "Create Brand" with no manufacturer selected despite every UI cue saying otherwise.

Additionally, the "surface the real server message" toast logic added in Plan 01 works only in the narrow case of whole-object Zod `.refine()` failures (`formErrors` populated) — for the overwhelmingly common case of a single required field being left blank, `fieldErrors` is populated instead and `formErrors` stays `[]`, so the toast still falls back to the generic "Validation failed" string rather than the specific field message the plan's must-have describes.

## Critical Issues

### CR-01: SKU submit-time validation blocks Save with no visible error for non-required fields

**File:** `apps/web/src/components/manage/SkuInlineEditor.tsx:184-200` (also `98-111`, `293-436`)
**File:** `apps/web/src/components/manage/ListingForm.tsx:261-265`

**Issue:** `validateAllRows()` runs `createSkuSchema.safeParse(buildSkuCandidate(row))` against the **entire** row and sets `allValid = false` on *any* failing field (line 190: `if (!result.success) allValid = false;`), but it only ever writes error messages into `skuErrors` for the four fields in `REQUIRED_SKU_FIELDS` (`name`, `price`, `msrp`, `quantity`, lines 191-196). The inputs for `sku`, `upc`, `size`, `casePack`, and `casesPerPallet` (lines 313-369) are never wrapped in `renderRequiredCell` and never get `onBlur` validation — there is no code path that can ever show an error for them.

`createSkuSchema` (packages/shared/src/schemas/inventory-sku.ts) requires `casePack`/`casesPerPallet` to be `.positive()` when provided, and caps `sku`/`size` at 100 chars and `upc` at 20 chars. Typing `0` into "Case Pack" — a completely natural value for a number input — fails `.positive()`, which fails the whole-row `safeParse`, which sets `allValid = false`. `ListingForm.onFormSubmit` then does:
```ts
const skusValid = skuEditorRef.current?.validateAll() ?? true;
if (!skusValid) return;
```
No toast, no field error, nothing — the user clicks "Save Changes"/"Create Listing" and literally nothing happens, with no way to discover why.

**Fix:** Either (a) extend `REQUIRED_SKU_FIELDS`/`renderRequiredCell` wiring to cover every schema-validated field (casePack, casesPerPallet, sku, upc, size), or (b) at minimum surface a toast when `validateAll()` returns false so the user knows *something* is wrong even if the specific cell isn't highlighted:
```ts
const skusValid = skuEditorRef.current?.validateAll() ?? true;
if (!skusValid) {
  toast.error('One or more SKU rows have invalid values. Please review the SKU section.');
  return;
}
```

### CR-02: "Manufacturer" is marked required in the UI but is not actually enforced by the schema

**File:** `apps/web/src/components/manage/BrandDialog.tsx:186-216`
**File:** `packages/shared/src/schemas/brand.ts` (referenced, not in review scope but load-bearing)

**Issue:** `BrandDialog` renders the manufacturer `Select` under `<RequiredLabel required>Manufacturer</RequiredLabel>`, triggers validation on close (`onOpenChange={(open) => { if (!open) void trigger('companyId'); }}`), and shows `<FieldError message={errors.companyId?.message} />` — every visual cue tells the admin this field is mandatory.

However, `createBrandSchema.companyId` is:
```ts
companyId: z.number().int().positive('Company is required').optional()
```
`.optional()` means `undefined` always passes validation — the `positive('Company is required')` message only fires if a value is present but not positive (e.g. a literal `0`), never for a value that was simply never selected. Because `defaultValues.companyId` is `undefined` and the admin can submit without ever opening the Select, `zodResolver` will happily pass, and `createBrand.mutate(data, ...)` fires with `companyId: undefined`, creating a brand with no manufacturer despite the asterisk and inline-error UI claiming otherwise. (Related, already-tracked: `deferred-items.md` records that `createBrandSchema` "does not reject a missing/negative companyId" for a different reason — the schema-level fix is out of scope for this phase, but this phase's UI actively compounds it by presenting the field as enforced.)

**Fix:** Either drop `.optional()` from `companyId` in `createBrandSchema` for the admin-create path (may need a separate schema variant since manufacturers self-serve without choosing a company), or, if the schema truly cannot change this phase, do not present the field with `RequiredLabel required` / wire blur validation that will never produce an error — that combination actively misleads the user.

## Warnings

### WR-01: "Real server message" toast rarely fires for the common single-field validation case

**File:** `apps/web/src/lib/api.ts:53-56`
**File:** `apps/web/src/hooks/useBrands.ts:69-75, 91-97, 110-116` (and equivalent blocks in `useCategories.ts`, `useListings.ts`)

**Issue:** `error.ts` always responds with `{ error: 'Validation failed', fieldErrors, formErrors }` for any `ZodError` — `error` is never `null`/`undefined`. That means in `apiFetch`:
```ts
const message = body?.error ?? body?.formErrors?.[0] ?? `API error: ...`;
```
the `body?.formErrors?.[0]` branch is unreachable dead code for every Zod-driven 400 (`body?.error` is always truthy), so `ApiError.message` is always the generic `"Validation failed"` string for these responses.

Separately, every hook's `onError` does:
```ts
error instanceof ApiError && error.formErrors.length
  ? error.formErrors[0]
  : error.message || 'Failed to create brand';
```
`err.flatten().formErrors` is populated only by issues with an *empty* path (whole-object `.refine()` failures). None of the schemas reviewed in this phase (`createBrandSchema`, `createCategorySchema`, `createListingSchema`, `createSkuSchema`) use object-level `.refine()` — every current validation issue (e.g. `name` required, `companyId` positive) has a non-empty path and lands in `fieldErrors`, not `formErrors`. So `error.formErrors.length` is `0` for essentially every real-world validation failure today, and the toast falls back to `error.message`, which — per the dead-code path above — is always `"Validation failed"`. The net effect: submitting a brand with a blank name shows "Validation failed" in the toast, not "Brand name is required" as the plan's must-have ("A failed mutation surfaces the real server message in a toast") implies.

**Fix:** Fall back to the first `fieldErrors` value when `formErrors` is empty:
```ts
const firstFieldError = Object.values(error.fieldErrors)[0]?.[0];
const message =
  error instanceof ApiError && (error.formErrors[0] ?? firstFieldError)
    ? (error.formErrors[0] ?? firstFieldError)
    : error.message || 'Failed to create brand';
```

### WR-02: SKU validation errors are misattributed after removing an unsaved row

**File:** `apps/web/src/components/manage/SkuInlineEditor.tsx:145-159` (`handleRemove`)

**Issue:** For a new (unsaved, no `id`) SKU row, `handleRemove` splices the row out of the array: `onChange(skus.filter((_, i) => i !== index))`. This shifts the array index of every subsequent row by one. `skuErrors`, however, is keyed by `` `${index}:${field}` `` and is never reconciled when the array is spliced — the error/validity state from the removed row's *successor* rows now lives under the wrong keys, so red borders / tooltips can appear on (or vanish from) the wrong row after deleting an earlier unsaved SKU.

**Fix:** When removing a new row, also purge/reindex `skuErrors` for indices greater than the removed one, e.g.:
```ts
setSkuErrors((prev) => {
  const next: Record<string, string> = {};
  for (const [key, msg] of Object.entries(prev)) {
    const [idxStr, field] = key.split(':');
    const idx = Number(idxStr);
    if (idx === index) continue;
    next[`${idx > index ? idx - 1 : idx}:${field}`] = msg;
  }
  return next;
});
```

### WR-03: `errorHandler` is not defensive against non-Error `err` values or invalid status codes

**File:** `apps/api/src/middleware/error.ts:13-17`

**Issue:** `err.status`, `err.statusCode`, and `err.message` are accessed unconditionally. If application code ever calls `next(err)`/throws with a non-object value (`null`, a string, a number — all valid JS `throw` targets), accessing `.status` on it throws a `TypeError` *inside the error handler itself*, which is the last line of defense in the Express pipeline. Separately, `status = err.status ?? err.statusCode ?? 500` uses `??`, so a falsy-but-defined `err.status` such as `0` is not replaced by the `500` default, and `res.status(0)` throws a `RangeError: Invalid status code`.

**Fix:**
```ts
const err2 = err && typeof err === 'object' ? err : { message: String(err) };
const rawStatus = err2.status ?? err2.statusCode;
const status = Number.isInteger(rawStatus) && rawStatus >= 100 && rawStatus <= 599 ? rawStatus : 500;
```

### WR-04: `CategoryDialog` display-order field cannot be cleared once touched

**File:** `apps/web/src/components/manage/CategoryDialog.tsx:135-145`

**Issue:** `displayOrder` is registered with `{ valueAsNumber: true }` and the schema field is `z.number().int().nonnegative().optional()`. If the user clears the input (e.g. select-all + delete), `valueAsNumber` produces `NaN`. Zod's `.optional()` only bypasses validation for `undefined`, not `NaN` — `z.number()` explicitly rejects `NaN` — so a cleared, "optional" field now fails validation and blocks the whole category form from submitting until the user re-types a number.

**Fix:** Coerce the empty string to `undefined` before validation, e.g. register with a custom `setValueAs`: `setValueAs: (v) => (v === '' ? undefined : Number(v))`.

### WR-05: Select components use display `name` as the value/lookup key instead of a stable id

**File:** `apps/web/src/components/manage/BrandDialog.tsx:190-213`
**File:** `apps/web/src/components/manage/ListingForm.tsx:356-382`

**Issue:** Both the "Manufacturer" select (BrandDialog) and the "Brand" select (ListingForm) use the record's `name` as the underlying `Select` value and resolve the chosen `id` via `companies.find(c => c.name === name)` / `brandOptions.find(b => b.name === name)`. If two companies (or two brands) share the same display name, `.find()` always returns the first match — selecting the second, differently-`id`'d record with a duplicate name will silently assign the wrong `companyId`/`brandId`.

**Fix:** Use a `String(id)` as the `Select`/`SelectItem` value instead of `name`, and look up the display label for rendering separately.

### WR-06: Non-numeric/garbage SKU quantity input is silently coerced to `0` before validation

**File:** `apps/web/src/components/manage/SkuInlineEditor.tsx:108` (`buildSkuCandidate`)
**File:** `apps/web/src/components/manage/SkuInlineEditor.tsx:526` (`buildSkuPayload`)

**Issue:** `quantity: Number(row.quantity) || 0` converts *before* the value reaches `createSkuSchema.safeParse`. Because `NaN` is falsy in JS, `Number(garbage) || 0` collapses any non-numeric quantity string to `0` — which is a **valid** value per `z.number().int().nonnegative()`. So invalid quantity input can never actually produce a "Quantity is invalid" validation error; it silently becomes `0` instead. The HTML `type="number"` input mitigates plain keyboard entry, but paste/autofill/some locales can still produce non-numeric strings that reach this code.

**Fix:** Distinguish "empty" from "invalid": `const n = row.quantity === '' ? undefined : Number(row.quantity); quantity: Number.isNaN(n) ? row.quantity : n` (or similar) so garbage input reaches the schema as a genuinely invalid value rather than being pre-coerced to a valid one.

## Info

### IN-01: Redundant payload reassignment with double type-casts in `ListingForm.onFormSubmit`

**File:** `apps/web/src/components/manage/ListingForm.tsx:285-316`

**Issue:** The shared branch (lines 285-302) already assembles `payload.skus`/`payload.images` in the `{create, update, delete}` shape when any operations exist. The `!listing` (create-mode) branch immediately below (lines 305-316) unconditionally rebuilds and overwrites both fields with a differently-shaped flat array, using `as unknown as ReturnType<typeof buildSkuPayload>` and `as unknown as typeof payload.images` to paper over the type mismatch between the two shapes. Functionally consistent today, but the double-assignment plus `as unknown as` casts make the two payload shapes (nested vs. flat) easy to break silently in a future edit since TypeScript can't check either cast.

**Fix:** Branch once on `listing` at the top of the function and build only the shape that's actually needed, avoiding the reassignment and the `unknown` casts.

### IN-02: `apiFetch`'s `formErrors[0]` fallback is unreachable for the current server contract

**File:** `apps/web/src/lib/api.ts:53-56`

**Issue:** As detailed in WR-01, `body?.error` is always populated by `error.ts` for every error response in this codebase (both `ZodError` and generic error branches always set `error: <string>`), so `body?.formErrors?.[0]` can never be reached — it is effectively dead code under the current server contract. Not harmful, but worth removing or documenting why it's kept as a defensive fallback for a hypothetical future response shape.
