<objective>
Restructure the API validation-error contract to Zod's flatten() shape and introduce a typed ApiError on the client so future field-level mapping is possible while keeping toast-only display now.

Purpose: This is the shared plumbing the whole phase depends on. It also fixes a pre-existing bug where apiFetch reads a nested `error.message` that the server never sends (RESEARCH Pitfall 6), so validation toasts currently show a generic "API error: 400".
Output: New server error body { error, fieldErrors, formErrors }; an ApiError class in api.ts; mutation hooks that prefer the real server message.
</objective>
