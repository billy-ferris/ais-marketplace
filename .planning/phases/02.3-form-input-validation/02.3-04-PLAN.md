    Client-side validation UX across the four manage forms plus a restructured API error contract:
    - ListingForm, BrandDialog, CategoryDialog: validate on blur, clear on valid change, catch-all + focus on submit, red asterisks on required fields, character counters on textareas, required selects error on blur.
    - SkuInlineEditor: per-cell blur validation with red border + hover tooltip; deleted rows skipped; submit-time catch-all blocks saving a listing with an invalid/untouched SKU row.
    - Server returns { error, fieldErrors, formErrors }; failed mutations show the real server message in a toast.
  </what-built>
  <how-to-verify>
    Start the app (`pnpm dev` or the project's usual dev command) and sign in.
    1. Categories → Create Category: click Name, tab out empty → inline "Category name is required"; type a name → error clears. Submit an empty dialog → error shows and Name focuses. Confirm Name label has a red `*`; icon/display order have none; no character counter appears.
    2. Brands → Create Brand: same blur/clear/submit-focus checks on Name (has `*`). Description textarea shows a live counter x/2000. As admin, the Manufacturer select shows a `*` and errors on blur when left empty.
    3. Listings → Create Listing: Name and Brand show `*`; Status shows none. Submit empty → Name & Brand error and the first invalid field is focused/scrolled to. Description textarea shows x/5000 counter. Categories section shows NO asterisk and does not block submit when empty.
    4. Status select (Create form): confirm status is pre-set to "Draft", shows NO asterisk, and cannot be emptied (no option clears it, no required error can fire). Confirm this handling is acceptable — this is the intentional deviation from CONTEXT.md's literal "(brand, status, categories)" wording because status has a schema default.
    5. In a listing, add a SKU row: leave Name empty and blur → red border + hover tooltip "SKU name is required". Enter "abc" in Unit Price and blur → red border + decimal-format tooltip. Fix values → borders/tooltips clear. Mark an existing SKU for deletion → it is never flagged invalid.
    6. SKU submit catch-all: add a SKU row and leave a required cell empty/invalid WITHOUT blurring it, then click Save/Create → the listing must NOT submit; the offending SKU cells show red border + tooltip; fixing them allows the save.
    7. Trigger a server rejection (e.g. submit a brand whose name collides or otherwise 400s) → the toast shows the real server message, not "API error: 400".
  </how-to-verify>
