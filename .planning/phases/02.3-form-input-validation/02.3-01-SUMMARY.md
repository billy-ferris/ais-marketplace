# Phase 02.3 Plan 01: Validation Error Contract Summary

Restructured the API validation-error body to Zod's `flatten()` shape (`{ error, fieldErrors, formErrors }`) and introduced a typed `ApiError` on the client, fixing a pre-existing bug where `apiFetch` read a nested `error.message` the server never sent — so validation toasts now show the real server message instead of a generic "API error: 400".

## What Was Built

### Task 1 — Error middleware to flatten() shape (TDD)
- `apps/api/src/middleware/error.ts`: ZodError branch now calls `const { fieldErrors, formErrors } = err.flatten();` and responds `400 { error: 'Validation failed', fieldErrors, formErrors }`. The `details` key is gone. `console.error` line and the non-Zod branch (500 masking, custom status/message passthrough) are unchanged.
- `apps/api/src/__tests__/middleware/error.test.ts`: new test asserting the flattened body shape, that `fieldErrors` values are string arrays, exact equality with `err.flatten()`, absence of `details`, plus masked-500 and custom-status non-Zod cases.
- TDD gates: RED commit `807ee43` (3 failing / 2 passing), GREEN commit `e3eacb0` (5 passing). No refactor needed.

### Task 2 — ApiError class + hook wiring
- `apps/web/src/lib/api.ts`: exports `class ApiError extends Error` with `status`, `fieldErrors: Record<string, string[]>`, `formErrors: string[]`. `apiFetch` parses the body once, computes `message = body?.error ?? body?.formErrors?.[0] ?? 'API error: …'` (reading the FLAT `error` string — fixes the nested-lookup bug) and throws `new ApiError(...)`.
- `useBrands.ts`, `useCategories.ts`, `useListings.ts`: import `ApiError`; every `onError` handler (3 + 3 + 8 = 14 mutations) prefers `error.formErrors[0]` when `error instanceof ApiError && error.formErrors.length`, else falls back to the original verbatim fallback string. Display stays toast-only.
