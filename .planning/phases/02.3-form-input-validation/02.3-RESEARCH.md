## User Constraints (from CONTEXT.md)

### Locked Decisions

**Validation Timing**
- Validate fields on blur (when user leaves the field)
- After an error appears, clear it on change as soon as the input becomes valid (instant positive feedback)
- Also validate all fields on submit as a final catch-all (catches untouched required fields)
- On submit with errors, scroll to and focus the first invalid field (react-hook-form `shouldFocusError`)

**Server Error Handling**
- Keep toast-only display of server validation errors for now (field-level mapping deferred to future phase)
- Restructure API error response format NOW: `{ error: 'Validation failed', fieldErrors: { name: 'message' }, formErrors: ['message'] }` — follows Zod's `flatten()` output shape
- HTTP status code only for categorization (no status in response body)
- Update `apiFetch` client utility to parse the new structured error format
- Create custom `ApiError` class extending `Error` with `fieldErrors`, `formErrors`, `status` properties
- Non-validation errors (500s, network failures) stay as toast — no special treatment needed
- Disable submit button with loading state during API call (react-hook-form `isSubmitting`)

- Validate on blur per field — consistent with main form timing
- Red border on invalid fields + tooltip on hover showing error message (compact for table rows)

## Summary

This is a UX-hardening phase over an already-working stack. Every target form already uses `useForm` + `zodResolver` with a consistent manual error-rendering pattern (`{errors.field && <p className="text-xs text-destructive">…</p>}`) and shared Zod schemas from `packages/shared/src/schemas/`. Three of the four forms (ListingForm, BrandDialog, CategoryDialog) are RHF-managed; the fourth (SkuInlineEditor) is a fully **uncontrolled, non-RHF array editor** with zero validation — it is the single largest delta in this phase.

The validation-timing changes are small, well-supported RHF config: `mode: 'onBlur'` + `reValidateMode: 'onChange'` gives "validate on blur, then clear-on-valid on change," and `shouldFocusError` (default `true`) already scrolls/focuses the first invalid field on submit. The API error restructure is a single-file change on the server (`apps/api/src/middleware/error.ts` — all routes funnel ZodErrors through it via `.parse()` + `next(err)`) plus a client change in `apps/web/src/lib/api.ts` to throw a new `ApiError` class instead of a bare `Error`.

Two structural landmines dominate planning: (1) **the web app has NO test infrastructure at all** (no vitest, no @testing-library/react, no jsdom, no test script) — the project's mandatory-test-coverage rule means a Wave 0 test-harness setup is required before form tests can exist; and (2) **this repo is a Base UI (`@base-ui/react`) shadcn install (`style: "base-nova"`), not Radix** — running `npx shadcn add form` would pull Radix-based components that conflict with the existing primitives. The recommendation is to **extend the existing raw register/errors pattern** with three tiny reusable helpers, not adopt shadcn's `Form`/`Field` wrapper.

**Primary recommendation:** Set `mode: 'onBlur'`/`reValidateMode: 'onChange'` on all three RHF forms; keep the existing raw RHF pattern and add small `RequiredLabel`, `FieldError`, and `CharCounter` helper components; retrofit SkuInlineEditor with a local per-row validation layer (blur-triggered `safeParse` against `createSkuSchema`, red border + Base UI Tooltip); restructure the error middleware to emit `flatten()`-shaped bodies and introduce an `ApiError` class in `api.ts`. Stand up web test infrastructure as Wave 0.

### SkuInlineEditor — `apps/web/src/components/manage/SkuInlineEditor.tsx`

### Shared plumbing
- **Error middleware** `apps/api/src/middleware/error.ts` — currently `{ error: 'Validation failed', details: string[] }`. Some routes also emit ad-hoc `res.status(400).json({ error: '...' })` (e.g., brands `companyId is required`) — those bypass the middleware and remain plain `{error}` (fine; client falls back to `formErrors`/toast).
- **API client** `apps/web/src/lib/api.ts` — currently throws `new Error(errorBody?.error?.message ?? ...)`. Note it reads `error.message` (nested) but the server sends flat `error` string — a **pre-existing mismatch** the ApiError refactor should fix. **Delta:** parse `{ error, fieldErrors, formErrors }`, throw `ApiError`.
- **Mutation hooks** `useBrands.ts`, `useCategories.ts`, `useListings.ts` — each has `onError: (error: Error) => toast.error(error.message || 'Failed to …')`. With `ApiError extends Error`, `error.message` still works (keep `error` string as the message). Optionally narrow with `instanceof ApiError` to prefer `formErrors[0]` or joined `fieldErrors`. Toast-only display is the locked decision, so minimal change needed.

## Architecture Patterns

### System Architecture Diagram (validation & error flow)
```
User types/blurs in field
        │
        ▼
RHF (mode:onBlur) ──► zodResolver ──► shared Zod schema ──► formState.errors
        │                                                        │
        │ reValidateMode:onChange (clear-on-valid)               ▼
        │                                          <FieldError/> + aria-invalid + red border
        ▼
Submit ──► handleSubmit validates ALL ──► errors? ──► shouldFocusError scrolls/focuses first
        │                                    │
        │ no errors                          └─► block submit
        ▼
mutation.mutate(data) ──► apiFetch ──► fetch()
        │                                 │
        │                          !response.ok
        │                                 ▼
        │                    parse { error, fieldErrors, formErrors }
        │                                 ▼
        │                        throw new ApiError(...)
        ▼                                 │
onError(err) ──► toast.error(err.message) ◄┘   (field mapping DEFERRED)

Server: route.parse(body) throws ZodError ──► next(err) ──► errorHandler
                                                                │
                                          err.flatten() → { error:'Validation failed',
                                                            fieldErrors:{f:'msg'}, formErrors:[...] }
```

### Pattern 1: RHF blur/revalidate timing (all three RHF forms)
```typescript
// Source: react-hook-form.com/docs/useform  [CITED]
const form = useForm<FormInput>({
  resolver: zodResolver(schema),
  mode: 'onBlur',          // validate a field when it loses focus
  reValidateMode: 'onChange', // AFTER an error shows, re-check on change → clears on valid
  // shouldFocusError: true is the DEFAULT — scrolls/focuses first invalid field on submit
  defaultValues: { /* ... */ },
});
```
- `mode: 'onBlur'` = first validation trigger is blur. `reValidateMode: 'onChange'` = once a field has an error, it re-validates on every change, so the error clears the instant the value becomes valid. This is exactly the locked "validate on blur, clear-on-valid on change" behavior. [CITED: react-hook-form.com/docs/useform]
- `handleSubmit` always validates all fields regardless of `mode`, satisfying the "catch untouched required fields on submit" requirement. [CITED]
- `shouldFocusError` defaults to `true` and focuses+scrolls the first errored field on failed submit — no extra work needed unless a field isn't natively focusable (Select — see Landmine). [CITED]

### Pattern 2: Small reusable helpers (extend existing pattern, do NOT adopt shadcn Form)
```typescript
// RequiredLabel — asterisk driven explicitly (schema-derived detection is brittle; see Required-field detection)
function RequiredLabel({ htmlFor, children, required }: { htmlFor?: string; children: React.ReactNode; required?: boolean }) {
  return (
    <Label htmlFor={htmlFor}>
      {children}{required && <span className="text-destructive" aria-hidden>*</span>}
    </Label>
  );
}

// FieldError — replaces the repeated {errors.x && <p .../>} blocks
function FieldError({ message }: { message?: string }) {
  return message ? <p className="text-xs text-destructive" role="alert">{message}</p> : null;
}

// CharCounter — only used under textareas
function CharCounter({ value, max }: { value: string; max: number }) {
  return <p className="text-xs text-muted-foreground text-right">{value?.length ?? 0}/{max}</p>;
}
```
Counter needs the live value: `const description = watch('description') ?? '';` then `<CharCounter value={description} max={5000} />`.

### Pattern 3: SkuInlineEditor per-cell validation with Base UI Tooltip
```typescript
// Local error state, keyed by "index:field"
const [skuErrors, setSkuErrors] = useState<Record<string, string>>({});

// Validate one field on blur against the shared schema (pick the single key)
function validateSkuField(index: number, field: keyof SkuFormData, raw: SkuFormData) {
  // Build the value in the schema's expected type (numbers are strings in the form)
  const candidate = {
    name: raw.name,
    price: raw.price,
    msrp: raw.msrp,
    quantity: Number(raw.quantity) || 0,
    casePack: raw.casePack ? Number(raw.casePack) : undefined,
    // ...only include the field(s) relevant; or safeParse whole row and read issues by path
  };
  const result = createSkuSchema.safeParse(candidate);
  setSkuErrors((prev) => {
    const key = `${index}:${field}`;
    const issue = result.success ? undefined
      : result.error.issues.find((i) => i.path[0] === field)?.message;
    return { ...prev, [key]: issue ?? '' };
  });
}
```
Tooltip on an invalid cell (Base UI — the trigger must wrap/render the input):
```tsx
// TooltipProvider must be mounted (see Landmine: Tooltip not mounted)
<Tooltip open={!!skuErrors[`${index}:price`] || undefined}>
  <TooltipTrigger render={
    <Input
      value={sku.price}
      aria-invalid={!!skuErrors[`${index}:price`]}
      className={skuErrors[`${index}:price`] ? 'border-destructive' : undefined}
      onChange={(e) => handleFieldChange(index, 'price', e.target.value)}
      onBlur={() => validateSkuField(index, 'price', sku)}
    />
  } />
  <TooltipContent>{skuErrors[`${index}:price`]}</TooltipContent>
</Tooltip>
```
Base UI `Tooltip.Trigger` uses a `render` prop (see `tooltip.tsx`) — pass the Input as the rendered element so hover/focus on the input shows the error. `aria-invalid` already triggers the red border via the Input's existing `aria-invalid:border-destructive` classes, so an explicit border class may be redundant — verify visually.

### Anti-Patterns to Avoid
- **Running `npx shadcn add form`** — pulls Radix-based `Form`/`Field` components incompatible with this Base UI (`style: base-nova`) repo. Confirmed: zero `@radix-ui` imports in `src`. Extend the existing pattern instead.
- **Converting SkuInlineEditor to `useFieldArray`** — a full rewrite of a working component (undo, `_deleted`, image upload). Out of scope; use a local validation layer.
- **Deriving "required" purely from Zod schema at runtime** — Zod v3 offers no clean public "is this field required" reflection; `.optional()` unwrapping is brittle and `.default()` fields read as optional in `z.input`. Hardcode asterisks per field (they're static and few). See Required-field detection.
- **Keystroke restriction on price/MSRP** — locked decision is blur-only decimal validation. Do NOT add `onChange` numeric filtering.

## Common Pitfalls

### Pitfall 1: Zod v3 `flatten()` returns arrays, decision wants strings
**What goes wrong:** The locked contract shows `fieldErrors: { name: 'message' }` (string values), but Zod v3 `flatten()` returns `fieldErrors: { name: ['message'] }` (string **arrays**) and `formErrors: string[]`.
**Why it happens:** `flatten()` groups multiple issues per field into an array.
**How to avoid:** In the middleware, either (a) send `flatten()` output as-is and treat `fieldErrors[field]` as `string[]` on the client (cleaner, future-proof), or (b) map to first-message strings: `Object.fromEntries(Object.entries(fieldErrors).map(([k,v]) => [k, v?.[0] ?? '']))`. **Recommend (a) arrays** — matches Zod, and the `ApiError` type should declare `fieldErrors: Record<string, string[]>`. Flag to planner that the CONTEXT example is illustrative shape, not exact typing. [VERIFIED: zod 3.25.76 flatten signature]

### Pitfall 2: `mutate()` isn't awaited → RHF `isSubmitting` never true
**What goes wrong:** BrandDialog/CategoryDialog call `createX.mutate(data)` synchronously inside the submit handler; RHF's `formState.isSubmitting` flips back to false immediately because the handler returns before the request finishes.
**Why it happens:** RHF ties `isSubmitting` to the promise returned by the submit callback.
**How to avoid:** These forms already disable the button with react-query `isPending` — that works today. If the decision's "use `isSubmitting`" is to be honored literally, switch handlers to `await mutateAsync(data)`. **Recommend:** keep `isPending` for dialogs (already correct) OR standardize on `mutateAsync`+`isSubmitting`; document choice. ListingForm receives `isSubmitting` as a prop from the mutation and is already fine.

### Pitfall 3: Controlled Select doesn't blur/focus like a native input
**What goes wrong:** `brandId`/`companyId`/`status` use Base UI `Select` + `watch`/`setValue`, not `register`. `mode: 'onBlur'` won't fire for them (no registered blur), and `shouldFocusError` can't focus a value that isn't a focusable registered field.
**Why it happens:** Selects are controlled via `setValue`, not RHF field registration.
**How to avoid:** Currently `setValue('brandId', id, { shouldValidate: true })` validates on change. For "error on blur if empty," add an `onBlur`/`onOpenChange(false)` handler that calls `form.trigger('brandId')`. For focus-on-submit, either register a hidden input or call `setFocus`/scroll manually — or accept that the inline error text appears (message shows even if focus lands elsewhere). Flag as Claude's-discretion detail.

### Pitfall 4: `z.input` vs `z.output` with `.default()` (already a known project gotcha)
**What goes wrong:** `createListingSchema.status` is `.optional().default('draft')`. `z.input` makes it optional; `z.output` makes it defined. ListingForm correctly types `useForm<z.input<...>>`. Adding required-asterisk logic must not treat defaulted fields as "required" (status has a default → no asterisk).
**Why it happens:** Default fields look optional on input, defined on output.
**How to avoid:** Hardcode asterisks; status gets none. Consistent with [Phase 02-05] decision already in STATE.md.

### Pitfall 5: Base UI Tooltip needs a mounted Provider
**What goes wrong:** `TooltipProvider` is defined in `tooltip.tsx` but is **not mounted anywhere** in `src`. SKU tooltips may not share delay/behavior or may warn.
**Why it happens:** No provider in the app tree yet.
**How to avoid:** Wrap the SkuInlineEditor (or app root) in `<TooltipProvider>`. Base UI tooltips can work per-instance, but mounting the provider is the intended pattern. Verify hover behavior in the table-row context.

### Pitfall 6: `apiFetch` reads `errorBody?.error?.message` but server sends flat `error`
**What goes wrong:** Pre-existing bug — client expects nested `error.message`, server sends `{ error: 'string' }`. Toasts currently fall back to the generic `API error: 400` string, losing the real message.
**How to avoid:** The `ApiError` refactor fixes this — read the flat `error` string as `message`. Verify toasts now show real server messages after the change.

## Required-field detection

**Recommendation: hardcode asterisks per field (do NOT derive from schema at runtime).** Rationale:
- Zod v3 has no stable public API to ask "is field X required?" Unwrapping `ZodOptional`/`ZodDefault` via `.shape` + `_def` is internal/brittle and breaks the `z.input` default case (status reads optional).
- Required fields are few and static per form; a hardcoded `required` prop on `RequiredLabel` is clear and reviewable.
- Fields the schemas mark required (source of truth for which get asterisks): listing `name`, `brandId`; brand `name` (+ `companyId` functionally for admin); category `name`; SKU `name`, `price`, `msrp`, `quantity`.

## API error restructure — concrete shapes

**Current server** (`error.ts`):
```typescript
res.status(400).json({ error: 'Validation failed', details: messages }); // messages: string[]
```
**New server** (using Zod v3 flatten):
```typescript
if (err instanceof ZodError) {
  const { fieldErrors, formErrors } = err.flatten();
  res.status(400).json({ error: 'Validation failed', fieldErrors, formErrors });
  return;
}
// fieldErrors: Record<string, string[]>, formErrors: string[]
```
**New client** (`api.ts`):
```typescript
export class ApiError extends Error {
  status: number;
  fieldErrors: Record<string, string[]>;
  formErrors: string[];
  constructor(message: string, status: number,
              fieldErrors: Record<string, string[]> = {}, formErrors: string[] = []) {
    super(message);
    this.name = 'ApiError';
    this.status = status;
    this.fieldErrors = fieldErrors;
    this.formErrors = formErrors;
  }
}
// in apiFetch, on !response.ok:
const body = await response.json().catch(() => null);
const message = body?.error ?? body?.formErrors?.[0]
  ?? `API error: ${response.status} ${response.statusText}`;
throw new ApiError(message, response.status, body?.fieldErrors ?? {}, body?.formErrors ?? []);
```
Mutation hooks keep `onError: (e) => toast.error(e.message)` (works since `ApiError extends Error`); optionally `if (e instanceof ApiError && e.formErrors.length) toast.error(e.formErrors[0])`.
