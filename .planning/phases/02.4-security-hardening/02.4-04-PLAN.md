<objective>
Push the data-integrity invariants from application code into Postgres: replace column-level `.unique()` on brand/category slugs with partial unique indexes scoped to non-deleted rows (D-09), add a partial unique index enforcing at most one primary image per listing (D-10), and declare explicit `onDelete` behavior on the flagged catalog FKs (D-13). Then run the schema push against the live dev DB.

Purpose: The reviews found application-level invariants (single primary image, live-slug uniqueness) are unenforceable against concurrency and code drift — the multi-primary bug arose exactly this way. Only Postgres constraints are race-free. The soft-delete pattern must be preserved, which is why plain `.unique()` (blocks slug reuse after delete) is wrong and a `WHERE deleted_at IS NULL` partial index is correct.

Output: Five modified schema files and a live DB with the new constraints applied.

NOTE: DB-level behavior (D-09/D-10/D-13) is NOT unit-testable in the mocked-db harness — it is verified by the manual DB UAT checklist in Plan 06. This plan is marked non-autonomous because `db:push` may present an interactive confirmation prompt for the constraint drop/create.
</objective>
