<objective>
Add the destructive-seed production guard (D-08) and make the R2 upload presign server-authoritative on content-type (D-06) and size (D-07).

Purpose: `seed.ts` performs mass `db.delete(...)` and Clerk user deletions; today nothing stops it running against a production `DATABASE_URL`. The presign endpoint currently signs any `contentType` and (as a presigned PUT) cannot cap size — a malicious client can upload an SVG (stored-XSS vector) or an oversized file to the public bucket. Both fixes are server-side and unit-testable in the mocked-db harness.

Output: An extracted, exported, unit-tested `assertNotProduction()` guard; a content-type allowlist; and a switch from presigned PUT to `createPresignedPost` with a `content-length-range` condition (new package `@aws-sdk/s3-presigned-post`).

NOTE: The presign response shape changes from `{ uploadUrl }` to `{ url, fields }`. The apps/web client that consumes it is updated separately in Plan 05 (manual-UAT — no web test harness).
</objective>

| client -> API (presign) -> R2 public bucket | Client dictates contentType and file size; the presigned policy is the only server-authoritative enforcement point |
| npm registry -> build | New third-party package enters the dependency tree |
