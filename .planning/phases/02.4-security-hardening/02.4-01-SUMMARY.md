# Phase 02.4 Plan 01: Access-Control & PII Redaction Hardening Summary

Closed the two Phase 01 IDOR criticals — broken access control on `GET /users/:id` and margin/PII over-exposure on company reads — by enforcing a self-or-admin deny-by-404 gate and a `redactCompany()` response guard, each proven by new Vitest coverage (RED then GREEN).

## What Was Built

### Task 1: Self-or-admin deny-by-404 on `GET /users/:id` (D-01, D-02)
- Read `{ userId, role }` from `getCurrentUser(req)` at the top of the handler.
- Replaced the lone `if (!user)` deny with a single combined branch:
  `if (!user || (role !== 'admin' && user.clerkId !== userId))` -> `404 { error: 'User not found' }`.
- Comparison is against `user.clerkId` (Clerk id string), never the numeric `users.id`.
- No separate 403 branch — a distinct 403 would reintroduce the enumeration oracle. `grep -n "403" apps/api/src/routes/users.ts` returns nothing.
- Behaviors covered by `users.test.ts` (5 tests): self->200, admin->200, other-non-admin->404, missing->404, invalid id->400.

### Task 2: Company field redaction (D-03, D-02)
- Added module-level `redactCompany(company, viewer)` where `viewer = { role, companyId }`.
- `isPrivileged = role === 'admin' || companyId === company.id`; privileged viewers get the full row, everyone else gets a row with `marginPercentage, contactName, phone, street, zip` dropped and `name, type, city, state` retained.
- `GET /` maps redaction across the array; `GET /:id` applies it to the single row after the existing 404. Added `getCurrentUser, getCompanyUser` to the imports (was only `requireAuth, requireRole`).
- Behaviors covered by `companies.test.ts` (7 tests): admin full row, owning-company full row, non-owner redacted (both list and detail), 404 for missing, 400 for invalid id.
