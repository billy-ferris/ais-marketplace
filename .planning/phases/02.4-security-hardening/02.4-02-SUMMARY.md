# Phase 02.4 Plan 02: Security Hardening (IDOR + Email Injection) Summary

**Closed cross-tenant brandId reassignment on PATCH /listings/:id and HTML/CRLF injection in approval emails via sink-side escaping/sanitization.**

## Performance

- **Duration:** ~12 min
- **Started:** 2026-07-01T11:04:00Z
- **Completed:** 2026-07-01T11:09:00Z
- **Tasks:** 2 (both TDD)
- **Files modified:** 4

## Accomplishments
- A manufacturer can no longer move a listing onto another company's brand via PATCH — the incoming `parsed.brandId` is now re-validated against `user.companyId` (returns 400), mirroring the existing POST check.
- Approval-email HTML body now escapes `listingName`, `rejectionReason`, and `recipientName`, blocking `<img onerror=...>`/`<script>` payloads.
- Approval-email subject strips CRLF, blocking `\r\nBcc:` header injection while leaving the subject un-HTML-escaped.
- Full API suite green (146 tests across 11 files).

## Decisions Made
- **Hoist over re-fetch:** `const user = await getCompanyUser(req)` moved to handler scope near the `role` read so it is in scope at the parse site, avoiding a second DB round-trip. Used `user?.companyId` at the new sink for TS null-safety (the manufacturer branch already 403s when companyId is absent).
- **Subject not HTML-escaped:** subjects are plain mail headers, so only CRLF sanitization applies; HTML-escaping would corrupt legitimate subject text.
- **No npm dependency:** the two escaping helpers are plain string replacements per the pattern map.
