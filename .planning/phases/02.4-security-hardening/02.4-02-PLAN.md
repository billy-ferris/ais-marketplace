<objective>
Fix two Phase 02.1 criticals: cross-tenant `brandId` reassignment on `PATCH /listings/:id` (a manufacturer could move a listing onto another company's brand) and HTML/CRLF injection in the approval-email sinks (`services/email.ts` interpolates user-controlled names/reasons raw into HTML and the subject line).

Purpose: `updateListingSchema = createListingSchema.partial()` includes `brandId`, but PATCH only ever validated the EXISTING brand — never the incoming one. Resend sends the `html` string verbatim with no auto-escaping, so an `<img onerror=...>` in a listing name or a `\r\nBcc:` in a subject reaches recipients. Both are handler-level and unit-testable in the mocked-db harness.

Output: Hardened `listings.ts` PATCH path (mirroring the existing POST ownership check) and `email.ts` (two dependency-free escaping helpers at every sink), plus new test cases.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> API (PATCH /listings/:id) | Manufacturer supplies an arbitrary brandId in the update payload; ownership must be re-validated server-side |
| stored data -> outbound email (Resend) | User-controlled names/reasons flow into an HTML body and a mail subject header; both are injection sinks |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.4-03 | Tampering / Elevation | PATCH /listings/:id brandId reassignment | mitigate | Re-validate the incoming `parsed.brandId` against `user.companyId`, mirroring the POST ownership check; hoist `user` to handler scope (D-12) |
| T-02.4-04 | Tampering / Spoofing | email.ts HTML body (phishing via unescaped names/reasons) | mitigate | `escapeHtml` at every HTML interpolation sink (listingName, rejectionReason, recipientName) (D-11) |
| T-02.4-05 | Tampering | email.ts subject header (CRLF injection) | mitigate | `sanitizeHeaderValue` strips `\r\n` from subject inputs (D-11) |
</threat_model>
