# Phase 02.4: Security Hardening - Research

**Researched:** 2026-07-01
**Domain:** Application security remediation (authorization scoping, injection escaping, DB constraint integrity, upload validation, operational safety) on an Express + TypeScript + Drizzle/Postgres API with a React/Vite client
**Confidence:** HIGH (all mechanics verified against installed code, the three REVIEW.md source files, and the installed dependency source; two non-trivial APIs verified against upstream)

## Summary

This phase is a **fixed-scope remediation** of Critical/high-severity findings from the retroactive reviews of phases 01, 02, and 02.1. There is no new product surface — every fix maps to a concrete finding with a recommended patch already written in the review files, further constrained by the D-01…D-13 decisions in CONTEXT.md. The research value here is (a) reconciling the review snippets with the *actual current code* (which has drifted in small but load-bearing ways), (b) nailing the two genuinely non-trivial mechanics (Drizzle partial unique indexes + presigned-upload size enforcement), and (c) mapping each fix class to the existing test harness so VALIDATION.md is accurate.

The codebase already contains the *correct* pattern for almost every fix somewhere adjacent to the bug: `GET /users/` is admin-gated (the `:id` IDOR is the gap), `POST /listings` validates `brandId` ownership (the `PATCH` path is the gap), `POST /listings` computes a single primary image (the schema constraint and `PATCH` path are the gaps). The remediation is overwhelmingly "mirror the existing-correct sibling and add a DB-level backstop," not "invent a new control."

**Primary recommendation:** Treat the API handler-level fixes (D-01/02/03, D-11, D-12, D-08, D-06) as unit-testable in the existing Vitest mocked-db harness and write real tests for each. Treat the DB-constraint fixes (D-09, D-10, D-13) as **not** unit-testable in that harness (db is mocked) — they require `drizzle-kit push` against a real Postgres plus manual DB-level UAT, and D-10 requires a **data-dedup migration step first** or index creation will fail. Treat the client changes (D-04/05 RoleGuard, and the D-07 PUT→POST client switch) as manual-UAT-only — `apps/web` has no test harness.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** `GET /users/:id` — enforce **self-or-admin** scope. A user may read their own record; admins may read any. Not admin-only (would break users loading their own profile by id).
- **D-02:** On any denied read (users *and* companies), return **404 Not Found**, not 403 — hides resource existence and prevents id enumeration.
- **D-03:** `GET /companies` (list and `:id`) — keep the endpoint **readable but field-redacted**. Strip `marginPercentage` and contact PII via a response-shaping function for callers who are neither admin nor the owning company. Do NOT block the endpoint outright (retailers legitimately browse manufacturer companies/brands).
- **D-04:** Wire the existing `RoleGuard` component into `apps/web/src/lib/router.tsx` (do not remove it). Route→role map:
  - `/manage/brands`, `/manage/listings/*` → **admin & manufacturer**
  - `/manage/categories`, `/manage/approvals/*` → **admin-only**
  - retailer → **no `/manage` access**
- **D-05:** Unauthorized deny UX = **`ForbiddenPage`** (RoleGuard's existing default `fallback`). No redirect. Client guard is UX-only; server API remains the real authorization boundary.
- **D-06:** Server-side content-type allowlist = **`image/jpeg`, `image/png`, `image/webp`**. Reject SVG and other types (note: this **excludes** `image/gif`, which the 02-REVIEW snippet had included — D-06 overrides). Enforced when presigning in `uploads.ts`.
- **D-07:** Max file size = **5 MB**, enforced server-side (e.g. presigned-POST content-length condition). Keep the client `ImageUploader` `maxSizeMB=5` default in sync.
- **D-08:** `db/seed.ts` must **hard-refuse (exit non-zero) when `NODE_ENV==='production'`**, with a clear error message, before any destructive delete or Clerk deletion runs. No override/escape-hatch env var.
- **D-09:** Soft-delete slug conflict → replace plain `unique()` on `slug` (brands/categories) with a **partial unique index scoped to non-deleted rows** (`WHERE deleted_at IS NULL`). Ref: 02-REVIEW CR-02.
- **D-10:** One-primary-image invariant → add a **partial unique index** on `brandListingImages` enforcing at most one `is_primary = true` per `listingId`. Ref: 02-REVIEW WR-05.
- **D-11:** Email injection → **HTML-escape** user-controlled input (`listingName`, `rejectionReason`) in the HTML body and **strip CRLF** from any value interpolated into the subject. Ref: 02.1-REVIEW CR-02.
- **D-12:** Cross-tenant `brandId` on `PATCH /listings/:id` → **mirror the POST ownership check** so a manufacturer cannot reassign a listing's `brandId` to another company's brand. Ref: 02.1-REVIEW CR-01.
- **D-13:** FK `onDelete` gaps → declare explicit `onDelete` behavior on the catalog FKs flagged in 02-REVIEW WR-04, consistent with the soft-delete strategy.

### Pattern 1: Handler-level auth context reads (the shape everything builds on)
**What:** All authorization data comes from two helpers in `apps/api/src/middleware/auth.ts`:
```ts
// Source: apps/api/src/middleware/auth.ts (verified current)
getCurrentUser(req)  // → { userId: clerkId | null, role: UserRole | null, sessionClaims }
getCompanyUser(req)  // async → { id, companyId, role, email, firstName } | null   (local DB row)
```
- `userId` is the **Clerk id** (a string), NOT the local integer `users.id`. To compare a fetched local user against the caller, compare `fetchedUser.clerkId === userId`.
- `role` comes from Clerk session claims (`sessionClaims.metadata.role`), no DB round-trip.
- `companyId` requires the async `getCompanyUser(req)` DB lookup — it is *not* in `getCurrentUser`.

### Pattern 2: "Self-or-admin, deny-by-404" (D-01/D-02, users.ts)
**What:** The 01-REVIEW snippet used a 403 on deny; **D-02 overrides that to 404** to prevent enumeration. Fetch first, then a *single* deny path returns 404 for both "not found" and "found but not yours."
```ts
// TARGET pattern for GET /users/:id  (adapts review snippet to D-02's 404-on-deny)
const { userId, role } = getCurrentUser(req);
const user = await db.query.users.findFirst({
  where: eq(users.id, id),
  with: { company: true },
});
// One 404 covers both "missing" and "not authorized" — no information leak.
if (!user || (role !== 'admin' && user.clerkId !== userId)) {
  res.status(404).json({ error: 'User not found' });
  return;
}
res.json(user);
```
**Pitfall:** Do NOT branch into a separate 403 for the "exists but not yours" case — that reintroduces the enumeration oracle D-02 is closing.

### Pattern 3: Companies response redaction (D-03)
**What:** Endpoint stays readable; a response-shaping helper strips `marginPercentage` and contact PII for callers who are neither `admin` nor the owning company. Determining "owning company" requires the caller's `companyId`, so the handler must call `getCompanyUser(req)` (not just `getCurrentUser`).
```ts
// Recommended helper shape (name at planner discretion)
type CompanyRow = typeof companies.$inferSelect;
function redactCompany(c: CompanyRow, viewer: { role: UserRole | null; companyId: number | null }): Partial<CompanyRow> {
  const isPrivileged = viewer.role === 'admin' || viewer.companyId === c.id;
  if (isPrivileged) return c;
  // Strip commercially-sensitive + contact PII. Keep name/type/city/state for browsing.
  const { marginPercentage, contactName, phone, street, zip, ...safe } = c;
  return safe;
}
```
- Apply in BOTH `GET /` (map over the array) and `GET /:id`.
- `GET /:id` for a non-existent id still returns 404 (existing behavior — keep it; D-02's deny-by-404 aligns, though for companies the endpoint isn't blocked, only redacted).
- **Decision point for planner:** exactly which fields count as "contact PII." Minimum per 01-REVIEW is `marginPercentage`, `contactName`, `phone`. Address fields (`street`,`city`,`state`,`zip`) are company address, less sensitive — recommend redacting `street`+`zip` (precise location) but keeping `city`/`state` (needed for browsing). Confirm with the copy/UX owner if unsure. `[ASSUMED]` on the exact PII field set beyond the review's three.

### Pattern 4: Mirror POST brandId ownership on PATCH (D-12, listings.ts)
**What:** `updateListingSchema` **includes `brandId`** (`createListingSchema.partial()`), and the current PATCH manufacturer block only validates the *existing* brand (line ~429), never the *new* `brandId` in the payload. Add the target-brand check after `updateListingSchema.parse`.
```ts
// After: const parsed = updateListingSchema.parse(listingData);  (~line 454)
if (role === 'manufacturer' && parsed.brandId !== undefined) {
  const [targetBrand] = await db
    .select({ companyId: brands.companyId })
    .from(brands)
    .where(and(eq(brands.id, parsed.brandId), isNull(brands.deletedAt)));
  if (!targetBrand || targetBrand.companyId !== user.companyId) {
    res.status(400).json({ error: 'Brand does not belong to your company' });
    return;
  }
}
```
**Scoping landmine (verified in current code):** the `user` object from `getCompanyUser(req)` is declared *inside* the earlier `if (role === 'manufacturer') { ... }` block (lines ~421–448) and is **out of scope** at the point where `parsed` becomes available (~line 454). The planner must either (a) hoist `user` to the handler scope, or (b) re-fetch `getCompanyUser(req)` in the new check. Prefer hoisting to avoid a second DB round-trip. Do NOT write the new check assuming `user` is already in scope — it compiles-fails or silently reads a different binding otherwise.

### Pattern 5: Email escaping (D-11, email.ts)
**What:** Resend sends the `html` string verbatim (no auto-escaping). Two helpers, two sinks:
```ts
// Source: 02.1-REVIEW CR-02 recommended fix (dependency-free)
function escapeHtml(s: string): string {
  return s.replace(/&/g, '&amp;').replace(/</g, '&lt;').replace(/>/g, '&gt;')
          .replace(/"/g, '&quot;').replace(/'/g, '&#39;');
}
function sanitizeHeaderValue(s: string): string {
  return s.replace(/[\r\n]/g, ' ');
}
```
- Apply `escapeHtml` to every user-controlled value interpolated into HTML: `listingName` (in `BODY_TEXT_MAP` and any `<strong>${name}</strong>`), `rejectionReason`, **and `recipientName`** (it comes from `users.firstName`, also user-controlled — the review focused on listingName/reason but recipientName is the same class of sink; escape it too).
- Apply `sanitizeHeaderValue` to the `SUBJECT_MAP` inputs (`listingName`) — CRLF stripping prevents header injection. `escapeHtml` is NOT wanted on the subject (subjects aren't HTML).
- **Ordering:** escape at the point of interpolation, not at the data source, so stored values remain clean and other consumers aren't affected.

### Pattern 6: Drizzle partial unique indexes (D-09, D-10)
**What:** Convert the third argument of `pgTable(...)` from the current object-only form to include a table-callback returning an index array. Remove the inline `.unique()` from the `slug` column.
```ts
// brands.ts (and identically categories.ts) — D-09
import { pgTable, /* ... */ uniqueIndex } from 'drizzle-orm/pg-core';
import { isNull } from 'drizzle-orm';           // isNull lives in the root export

export const brands = pgTable('brands', {
  // ...
  slug: varchar({ length: 255 }).notNull(),      // <-- remove `.unique()`
  // ...
}, (table) => [
  uniqueIndex('brands_slug_active_unique')
    .on(table.slug)
    .where(isNull(table.deletedAt)),
]);
```
```ts
// brand-listing-images.ts — D-10
import { pgTable, /* ... */ uniqueIndex } from 'drizzle-orm/pg-core';
import { sql } from 'drizzle-orm';

export const brandListingImages = pgTable('brand_listing_images', {
  // ...
}, (table) => [
  uniqueIndex('brand_listing_images_one_primary_per_listing')
    .on(table.listingId)
    .where(sql`${table.isPrimary} = true`),
]);
```
- **VERIFIED:** the `.where(condition: SQL)` method on the index builder exists in installed `drizzle-orm@0.45.1` (`node_modules/drizzle-orm/pg-core/indexes.js` line 92) — this is not a training-data guess.
- Import note: the review snippet imported `isNull` from `drizzle-orm/pg-core`; the existing codebase imports `isNull` from `drizzle-orm` (root). Both re-export it, but **match the existing convention** (`drizzle-orm` root) to keep imports consistent. `uniqueIndex` is only in `drizzle-orm/pg-core`.

### Pattern 7: FK onDelete strategy (D-13)
**What:** Five FKs currently omit `onDelete` (Postgres default `NO ACTION`). Choose explicitly. Given the app uses **soft-delete everywhere and never hard-deletes in normal flow**, the onDelete branch essentially only fires from `seed.ts` cleanup or future maintenance code. Recommended, aligned to ownership semantics:

| FK | File | Recommended onDelete | Rationale |
|----|------|---------------------|-----------|
| `brands.companyId → companies.id` | brands.ts | `restrict` | A brand should never silently vanish because a company row is removed; fail loudly |
| `inventorySkus.listingId → brandListings.id` | inventory-skus.ts | `cascade` | SKUs are owned parts of a listing |
| `brandListingImages.listingId → brandListings.id` | brand-listing-images.ts | `cascade` | Images are owned parts of a listing |
| `listingCategories.listingId → brandListings.id` | listing-categories.ts | `cascade` | Join row is meaningless without its listing |
| `listingCategories.categoryId → categories.id` | listing-categories.ts | `restrict` | Don't let a category hard-delete quietly strip listings' category tags |

This is Claude's-discretion territory (D-13 says "consistent with the soft-delete strategy"). The above is the recommendation; a defensible alternative is `restrict` on all five (surfaces every accidental hard-delete). Avoid `set null` — all five columns are `NOT NULL`, so `set null` would error. `[ASSUMED]` on the exact cascade-vs-restrict split; the planner should state the chosen policy explicitly so the plan-checker can verify it.

### Pattern 8: Presigned upload validation (D-06 + D-07, uploads.ts)
**D-06 (content-type)** is trivial and needs no client change — the presigned URL already signs `ContentType`, so rejecting a non-allowlisted `contentType` before signing is fully server-authoritative:
```ts
const ALLOWED_CONTENT_TYPES = ['image/jpeg', 'image/png', 'image/webp']; // D-06: no gif, no svg
if (!ALLOWED_CONTENT_TYPES.includes(contentType)) {
  res.status(400).json({ error: 'Unsupported content type' });
  return;
}
```
**D-07 (5 MB size)** cannot be enforced by a presigned PUT. Recommended: switch the presign endpoint to `createPresignedPost` with a `content-length-range` condition.
```ts
// Source: @aws-sdk/s3-presigned-post — verified R2-compatible (Cloudflare docs)
import { createPresignedPost } from '@aws-sdk/s3-presigned-post';
const MAX_UPLOAD_BYTES = 5 * 1024 * 1024;
const { url, fields } = await createPresignedPost(s3Client, {
  Bucket: process.env.R2_BUCKET_NAME!,
  Key: key,
  Conditions: [
    ['content-length-range', 1, MAX_UPLOAD_BYTES],
    ['eq', '$Content-Type', contentType],
  ],
  Fields: { 'Content-Type': contentType },
  Expires: 600,
});
res.json({ url, fields, publicUrl, key });
```
**Client impact (apps/web, manual-UAT only):** presigned POST changes the client from `fetch(uploadUrl, { method:'PUT', body:file })` to a **multipart POST** — build a `FormData`, append every entry from `fields` first, then append `file` **last**, and POST to `url`. This touches `apps/web/src/hooks/useUpload.ts` (and the response shape returned by the API). `ImageUploader`'s `maxSizeMB=5` stays as the client pre-check per D-07. Also apply 02.1-REVIEW WR-02 while here: check `uploadRes.ok` before returning `publicUrl`.

### Anti-Patterns to Avoid
- **403-on-deny for reads** — reintroduces the enumeration oracle (D-02 mandates 404).
- **Client-only upload validation** — the whole point of D-06/D-07 is server authority; `accept="image/*"` is a picker hint, bypassable via drag-drop (02-REVIEW WR-03).
- **Escaping at the data source** instead of at the HTML sink — corrupts stored data and other consumers.
- **Adding an override env var to the seed guard** — explicitly forbidden by D-08.
- **Creating the one-primary index without deduping first** — see Pitfall 1.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Upload size limit | Custom byte-counting proxy / post-upload polling | `createPresignedPost` `content-length-range` | R2/S3 enforce it at ingest; server signs the constraint |
| "At most one primary image" | Application-level check-then-update | Partial unique index (D-10) | DB is the only race-free enforcement point; app checks miss concurrent writes (this is exactly how the bug arose on the PATCH path) |
| Slug uniqueness among live rows | App-level "is this slug taken?" query | Partial unique index `WHERE deleted_at IS NULL` (D-09) | Atomic; also fixes the soft-delete reuse bug |
| HTML escaping | (borderline) — but keep the 5-char helper | Inline `escapeHtml` from review | Zero-dependency, sufficient for this text-only sink; a full sanitizer is overkill |

**Key insight:** Three of the four fixes are *database constraints*, because the reviews found that application-level invariants (single primary image, slug uniqueness) are unenforceable against concurrency and code drift. Push the invariant into Postgres.

## Common Pitfalls

### Pitfall 1: One-primary index creation fails on dirty data
**What goes wrong:** `drizzle-kit push` tries to `CREATE UNIQUE INDEX ... WHERE is_primary = true` and Postgres rejects it with a `duplicate key value violates unique constraint` because pre-existing rows already have multiple primaries per listing.
**Why it happens:** The PATCH image path never enforced single-primary (02.1-REVIEW WR-03).
**How to avoid:** Add a **data-cleanup task that runs before (or as part of) the migration**: for each `listing_id` with >1 primary, keep one (lowest `id` or lowest `display_order`) and set the others `is_primary=false`. Verify with a pre-check query. This is a distinct plan task from the schema edit.
**Warning signs:** `db:push` aborts with a unique-violation naming `brand_listing_images_one_primary_per_listing`.

### Pitfall 2: `drizzle-kit push` interactivity / constraint-drop
**What goes wrong:** Push must DROP the existing `brands_slug_unique` / `categories_slug_unique` constraints and CREATE the partial indexes. `push` can prompt for confirmation on statements it deems potentially destructive, and interactive prompts (`-i`) aren't supported in the automation harness.
**Why it happens:** Removing a column-level `.unique()` and replacing it with a table index is a constraint change.
**How to avoid:** Run `pnpm --filter @ais/api db:push` and review the printed SQL. Dropping a unique constraint is not row-data loss, so it typically applies without a destructive-prompt; if a prompt appears, run interactively in a real terminal (outside the agent) or generate+inspect via `db:generate` first. Note the project has **no `drizzle/` migrations directory** today — it operates in push mode; do not introduce generated-migration files unless the planner deliberately switches workflows.
**Warning signs:** `db:push` hangs awaiting input; a constraint named `*_slug_unique` still present after push.

### Pitfall 3: `user` binding out of scope in the PATCH brandId check
**What goes wrong:** The new D-12 check references `user.companyId`, but `user` is block-scoped to the earlier manufacturer branch and undefined at the parse site.
**How to avoid:** Hoist `const user = await getCompanyUser(req)` to the handler top of the manufacturer path (or re-fetch). See Pattern 4.
**Warning signs:** TS `Cannot find name 'user'` or a reference to a different `user`.

### Pitfall 4: Seed guard placed after `db` initialization
**What goes wrong:** `seed.ts` initializes `drizzle(process.env.DATABASE_URL!)` and the Clerk client at **module top-level** (lines ~16-17). A guard added only inside `main()` still constructs a production DB client on import, and makes the guard hard to unit-test (importing the module has side effects / throws on missing env).
**How to avoid:** Extract a pure `assertNotProduction()` function (`if (process.env.NODE_ENV === 'production') { console.error(...); process.exit(1); }`) and call it as the **first statement of `main()`**, before any `db.delete(...)`/`cleanClerkUsers()`. Keeping it a small pure function makes it unit-testable (mock `process.exit`) without importing the whole seed side-effect chain. Per D-08: no override var.
**Warning signs:** Guard runs but destructive deletes already fired; test can't import the guard without a live `DATABASE_URL`.

### Pitfall 5: R2 presigned-POST field ordering / Content-Type signature mismatch
**What goes wrong:** With `['eq', '$Content-Type', contentType]`, the client MUST send exactly that Content-Type, and in a multipart POST the `file` field must be appended **after** all policy `fields`, or R2 returns 403 `SignatureDoesNotMatch` / policy-condition failures.
**How to avoid:** In `useUpload`, append every `fields` entry to `FormData` first, then append the file last; do not set a manual `Content-Type` on the `fetch` (let the browser set the multipart boundary). Confirm R2 bucket CORS allows POST (existing config allowed PUT — POST may need adding).
**Warning signs:** Uploads 403 on POST though PUT worked; "Policy condition failed: content-length-range."
