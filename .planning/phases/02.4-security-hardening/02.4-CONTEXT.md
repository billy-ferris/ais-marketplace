# Phase 02.4: Security Hardening - Context

**Gathered:** 2026-07-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Remediate the Critical and high-value Warning findings surfaced by the retroactive code reviews of phases 01, 02, and 02.1 so the app is safe to deploy. Fixes span: API authorization scoping (IDOR on `GET /users/:id` and `GET /companies`; cross-tenant `brandId` on `PATCH /listings/:id`), injection escaping (email HTML/subject sinks), a destructive-seed environment guard, DB constraint corrections (soft-delete slug conflict, one-primary-image invariant, FK `onDelete`), R2 upload content-type/size validation, and wiring the currently-dead client-side `RoleGuard`.

**Boundary anchor:** The set of findings is fixed by ROADMAP.md and the `NN-REVIEW.md` files. This phase fixes *how* those findings are remediated; it does not add new features. Verification path is API-side tests + manual UAT — there is no web test harness (see `[[phase-02-3-web-tests-deferred]]`).

</domain>

<decisions>
## Implementation Decisions

### Read-access authorization (Phase 01 IDOR criticals)
- **D-01:** `GET /users/:id` — enforce **self-or-admin** scope. A user may read their own record; admins may read any. Not admin-only (would break users loading their own profile by id).
- **D-02:** On any denied read (users *and* companies), return **404 Not Found**, not 403 — hides resource existence and prevents id enumeration.
- **D-03:** `GET /companies` (list and `:id`) — keep the endpoint **readable but field-redacted**. Strip `marginPercentage` and contact PII via a response-shaping function for callers who are neither admin nor the owning company. Do NOT block the endpoint outright (retailers legitimately browse manufacturer companies/brands).

### Client role gating (RoleGuard wiring)
- **D-04:** Wire the existing `RoleGuard` component into `apps/web/src/lib/router.tsx` (do not remove it). Route→role map:
  - `/manage/brands`, `/manage/listings/*` → **admin & manufacturer** (manufacturers self-serve; API already scopes by company)
  - `/manage/categories`, `/manage/approvals/*` → **admin-only** (taxonomy + approval workflow)
  - retailer → **no `/manage` access**
- **D-05:** Unauthorized deny UX = **`ForbiddenPage`** (RoleGuard's existing default `fallback`). No redirect. Client guard is UX-only; the server API remains the real authorization boundary.

### Upload validation (server-side is authoritative)
- **D-06:** Server-side content-type allowlist = **`image/jpeg`, `image/png`, `image/webp`**. Reject SVG and other types. Enforced when presigning in `uploads.ts` (reject unknown `contentType` before issuing a presigned URL).
- **D-07:** Max file size = **5 MB**, enforced server-side (e.g. presigned-POST content-length condition). Keep the client `ImageUploader` `maxSizeMB=5` default in sync so client and server agree.

### Destructive seed guard
- **D-08:** `db/seed.ts` must **hard-refuse (exit non-zero) when `NODE_ENV==='production'`**, with a clear error message, before any destructive delete or Clerk deletion runs. No override/escape-hatch env var — a seed script never needs to run against production.

### Delegated to planner — follow the review recommendations directly (no product decision needed)
- **D-09:** Soft-delete slug conflict (brands/categories) → replace plain `unique()` on `slug` with a **partial unique index scoped to non-deleted rows** (`WHERE deleted_at IS NULL`). Preserves the soft-delete pattern while allowing slug reuse after delete. Ref: 02-REVIEW CR-02.
- **D-10:** One-primary-image invariant → add a **partial unique index** on `brandListingImages` enforcing at most one `is_primary = true` per `listingId`. Ref: 02-REVIEW.
- **D-11:** Email injection (`services/email.ts`) → **HTML-escape** user-controlled input (`listingName`, `rejectionReason`) in the HTML body and **strip CRLF** from any value interpolated into the subject. Ref: 02.1-REVIEW.
- **D-12:** Cross-tenant `brandId` on `PATCH /listings/:id` → **mirror the POST ownership check** so a manufacturer cannot reassign a listing's `brandId` to another company's brand. Ref: 02.1-REVIEW.
- **D-13:** FK `onDelete` gaps → declare explicit `onDelete` behavior on the catalog FKs flagged in 02-REVIEW, consistent with the soft-delete strategy.

### Claude's Discretion
- Exact error-message copy, the shape/name of the response-shaping (redaction) helper for companies, the Drizzle migration mechanics for the partial indexes, and the precise presigned-upload size-enforcement technique are left to the planner/executor.
- Slug-generation TOCTOU race: address opportunistically if the partial-index work makes it low-cost (a unique-violation retry/catch); otherwise the partial index is the primary guard.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `apps/web/src/components/shared/RoleGuard.tsx` — already built and functional; reads `useRole()`, renders `ForbiddenPage` on deny. Just needs wiring into the router. Accepts `allowedRoles: UserRole[]` and an optional `fallback`.
- `apps/api/src/middleware/auth.ts` `requireAuth()` — the custom middleware returning 401 JSON; the authz fixes layer role/ownership checks on top of it (do not replace it).
- `apps/web/src/components/shared/ImageUploader.tsx` — already has `accept='image/*'` and `maxSizeMB=5` client defaults; server limits (D-06/D-07) should agree with these.
- `ForbiddenPage`, `useRole` hook, `UserRole` / `ALL_ROLES` from `@ais/shared` — existing role primitives.

### Established Patterns
- Soft-delete via `deletedAt` is the established pattern for brands/categories — fixes must preserve it (drives the partial-unique-index approach, D-09).
- List endpoints were already deliberately admin-gated in Phase 02 — the IDOR gaps are on single-resource/detail reads, so fixes are localized to those handlers.
- POST `/listings` already validates `brandId` ownership — PATCH must mirror it (D-12), not invent a new check.

### Integration Points
- `apps/api/src/routes/users.ts` (GET `/users/:id`), `apps/api/src/routes/companies.ts` (list + `:id`), `apps/api/src/routes/listings.ts` (PATCH `brandId`), `apps/api/src/services/email.ts`, `apps/api/src/routes/uploads.ts` (presign), `apps/api/src/db/seed.ts`, catalog schema (`brands`/`categories` slug, `brandListingImages.isPrimary`, FKs), `apps/web/src/lib/router.tsx`.

</code_context>

<specifics>
## Specific Ideas

- Deny-by-404 (not 403) is a deliberate anti-enumeration choice and applies consistently to both users and companies denies.
- Companies stay browsable — the fix is redaction of `marginPercentage` + contact PII, not access removal — because retailers legitimately view manufacturer companies/brands.
- Seed guard has intentionally **no** override env var; simplicity over flexibility to eliminate the misconfiguration foot-gun.

</specifics>

## Deferred Ideas

None of the above expand this phase's scope — they are lower-priority items explicitly noted so they aren't lost.
