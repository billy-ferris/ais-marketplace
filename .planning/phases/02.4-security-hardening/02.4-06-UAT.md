## Gaps

### Gap 1 — R2 does not support presigned POST (blocking)
- **Symptom:** Uploading a valid JPEG fails with a browser "CORS error" against `*.r2.cloudflarestorage.com`, despite a correct bucket CORS policy (`AllowedOrigins: http://localhost:5173`, `AllowedMethods: [GET, PUT, POST]`, `AllowedHeaders: [*]`).
- **Root cause:** Cloudflare R2's S3-compatible API does **not** implement the POST Object operation. `@aws-sdk/s3-presigned-post` `createPresignedPost` (shipped in Plan 03 for D-07, consumed in Plan 05) is unsupported by R2. R2 rejects the POST; its error response omits `Access-Control-Allow-Origin`, so the browser surfaces it as CORS.
  - Source: https://developers.cloudflare.com/r2/api/s3/presigned-urls/ ("presigned URLs support GET/PUT; POST is not supported").
  - This contradicts 02.4-RESEARCH.md's unverified assumption (line ~435) that "R2 supports presigned POST."
- **Remediation:** Revert the upload presign from POST back to presigned **PUT** (`@aws-sdk/s3-request-presigner` + `PutObjectCommand`), which Plan 03 removed. Enforce D-07 size via the research's documented fallback: sign an **exact `ContentLength`** (server rejects >5 MB before signing; R2 rejects at ingest on byte mismatch) and sign `Content-Type` to preserve the D-06 allowlist pin. Client `useUpload` reverts to `PUT {url}` with the file body + Content-Type/Content-Length headers.
- **Affected files:** `apps/api/src/routes/uploads.ts`, `apps/api/src/__tests__/routes/uploads.test.ts`, `apps/api/package.json`, `pnpm-lock.yaml`, `apps/web/src/hooks/useUpload.ts`.

### Gap 2 — Upload errors not surfaced in the UI
- **Symptom:** Selecting an SVG is correctly rejected server-side (400 "Unsupported content type", D-06 works), but the user sees no feedback — the error only appears in the browser console.
- **Root cause:** `ImageUploader.tsx` has an `error` display but only sets it for its own client-side size pre-check. The actual upload runs in the parent via `onChange(file)` → `useUpload().uploadFile`, whose thrown `ApiError` is not caught/surfaced back into the component.
- **Remediation:** Wire `uploadFile`'s rejection into the UI — add an `error`/`onError` path to `ImageUploader` (or catch in the consuming form and pass an error prop) so server rejections and upload failures render in the existing `text-destructive` slot. Optionally add a client-side content-type pre-check mirroring the server allowlist.
- **Affected files:** `apps/web/src/components/shared/ImageUploader.tsx` and its consuming form(s) (apps/web — no test harness; manual UAT).
