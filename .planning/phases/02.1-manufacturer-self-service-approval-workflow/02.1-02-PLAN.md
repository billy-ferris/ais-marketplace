  - Gets clerkId from `getCurrentUser(req).userId`
  - Queries `db.query.users.findFirst({ where: eq(users.clerkId, clerkId!), columns: { id: true, companyId: true, role: true, email: true, firstName: true } })`
  - Returns the user record (or null if not found)
- This helper centralizes the companyId lookup pattern described in RESEARCH.md Pattern 1

- After getting the search/companyId/page/limit params, check role via `getCurrentUser(req)`
- If role is `manufacturer`: call `getCompanyUser(req)` to get companyId, force-add `eq(brands.companyId, user.companyId)` to conditions (overriding any companyId query param -- manufacturer can only see their company's brands)
- If role is `admin`: existing logic unchanged (companyId filter is optional from query param)

- Change to `requireRole('admin', 'manufacturer')`

**GET `/:id`:**

  - Force status to `'draft'` regardless of what was submitted (per user decision: manufacturer creates as draft, then explicitly submits for review)
- If role is `admin`:
  - Existing behavior -- can set any status. Per APRV-07, admin bypasses approval.

**PATCH `/:id`:**
- Change to `requireRole('admin', 'manufacturer')`
- If role is `manufacturer`:
  - Get companyUser, verify listing belongs to company (via brand's companyId)
  - Check listing current status: if `pending_approval`, return 403 "Cannot edit listing while pending approval" (per user decision, locked while pending)
  - If listing is `rejected`, allow editing
  - Manufacturer cannot change status via PATCH (status changes happen via dedicated endpoints below)
  - Strip `status` field from request body if manufacturer role
- If role is `admin`: existing behavior

**DELETE `/:id`:**
- Keep as `requireRole('admin')` only

**Add new approval workflow endpoints to listings.ts:**

**POST `/:id/submit`** - Manufacturer submits for review:
- `requireAuth(), requireRole('admin', 'manufacturer')`
- If role is manufacturer:
  - Get companyUser, verify listing belongs to company
  - Check current status is `draft` or `rejected`. If not, return 400 "Can only submit listings in draft or rejected status"
  - Update listing: SET status='pending_approval', rejectionReason=null, updatedAt=now() WHERE id=:id AND status IN ('draft', 'rejected')
  - If zero rows affected, return 409 "Listing status has already changed"
  - Call `notifyApprovalEvent({ type: 'listing_submitted', listingId, listingName, recipientUserIds: [all admin user IDs] })`
    - To get admin user IDs: `db.select({ id: users.id }).from(users).where(eq(users.role, 'admin'))`
  - Return updated listing
- If role is admin: admin can also submit (useful for moving a listing to pending_approval), same logic but skip company check

**POST `/:id/approve`** - Admin approves listing:
- `requireAuth(), requireRole('admin')`
- Check current status is `pending_approval`. If not, return 409 "Listing is not pending approval"
- Update: SET status='active', rejectionReason=null, updatedAt=now() WHERE id=:id AND status='pending_approval'
- If zero rows affected, return 409 "Listing has already been processed"
- Look up the listing creator: find the brand's companyId, then find users with that companyId to notify
  - Query: find the brand for this listing -> get companyId -> find all users where companyId matches
- Call `notifyApprovalEvent({ type: 'listing_approved', listingId, listingName, recipientUserIds: [company users] })`
- Return updated listing

**POST `/:id/reject`** - Admin rejects listing:
- `requireAuth(), requireRole('admin')`
- Parse request body: `{ reason: string }` -- reason is required, min length 1
- Check current status is `pending_approval`. If not, return 409
- Update: SET status='rejected', rejectionReason=reason, updatedAt=now() WHERE id=:id AND status='pending_approval'
- If zero rows affected, return 409
- Look up company users (same as approve)
- Call `notifyApprovalEvent({ type: 'listing_rejected', listingId, listingName, recipientUserIds: [company users], rejectionReason: reason })`
- Return updated listing

**POST `/:id/archive`** - Manufacturer archives active listing:
- `requireAuth(), requireRole('admin', 'manufacturer')`
- If manufacturer: verify listing belongs to company, check status is `active`
- Update: SET status='archived', updatedAt=now() WHERE id=:id AND status='active'
- Return updated listing
- (Reactivation per user decision requires resubmission: manufacturer edits then calls /:id/submit to go back to pending_approval. The submit endpoint already handles `archived` would need to be added to accepted statuses, or manufacturer first moves to draft. Per user decision: "archived -> pending_approval (manufacturer reactivates, requires re-approval)". So the submit endpoint should also accept `archived` status as a valid source.)

IMPORTANT: For all approval endpoints, use optimistic locking by including the expected current status in the WHERE clause (per RESEARCH.md Pitfall 3). This prevents race conditions.

IMPORTANT: Category routes (`apps/api/src/routes/categories.ts`) remain admin-only -- do NOT modify. Per user decision MFSS-03, manufacturers can assign existing categories to listings but cannot create, edit, or delete categories. The category assignment happens through the listing POST/PATCH body's `categoryIds` field.
