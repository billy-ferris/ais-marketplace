# Phase 2.1: Manufacturer Self-Service & Approval Workflow - Research

**Researched:** 2026-03-09
**Domain:** Role-based multi-tenant CRUD, approval state machines, notification systems, transactional email
**Confidence:** HIGH

## Summary

Phase 2.1 extends the existing admin-only catalog management into a multi-tenant system where manufacturers can manage their own brands and listings, gated by an admin approval workflow. The phase also introduces an in-app notification inbox and transactional email via Resend. All three domains (self-service, approval, notifications) are well within the capabilities of the existing stack and require no new architectural patterns -- they extend existing Express routes, Drizzle schema, React pages, and TanStack Query hooks.

The primary technical challenges are: (1) modifying existing admin-only routes to support dual role access with company scoping, (2) extending the listing status enum with `pending_approval` and `rejected` states plus a `rejection_reason` column, (3) designing a notifications table and polling mechanism, and (4) integrating Resend for email delivery. None of these require new libraries beyond `resend`.

**Primary recommendation:** Extend existing routes/pages with role-aware scoping rather than creating parallel manufacturer-specific endpoints. Add a `notifications` table with typed events, a `/api/notifications` route, and use TanStack Query polling (30-60s interval) for the unread count badge.

### Pattern 1: CompanyId Resolution from Session
**What:** The current Clerk session token contains `metadata.role` but NOT `companyId`. To get companyId, query the local `users` table using `clerkId` from the session.
**When to use:** Every manufacturer-scoped route handler
**Example:**
```typescript
// In route handler:
const { userId } = getCurrentUser(req);
const user = await db.query.users.findFirst({
  where: eq(users.clerkId, userId!),
  columns: { companyId: true, role: true },
});
// Then use user.companyId to scope queries
```

**Key insight:** The existing `getCurrentUser()` returns `userId` (clerkId), `role`, and `sessionClaims`. It does NOT return `companyId` because companyId is in the local DB, not in Clerk's JWT. Each route that needs company scoping must do a DB lookup. This is a single indexed query (clerkId has unique index) and can be memoized per request if needed. Alternatively, add a `requireCompanyUser()` middleware that attaches `req.companyUser` to the request.

### Pattern 2: Dual-Role Route Access
**What:** Existing routes use `requireRole('admin')`. For manufacturer self-service, change to `requireRole('admin', 'manufacturer')` and branch logic based on role.
**When to use:** brands GET/POST/PATCH, listings GET/POST/PATCH
**Example:**
```typescript
router.get('/', requireAuth(), requireRole('admin', 'manufacturer'), async (req, res, next) => {
  const { role } = getCurrentUser(req);

  if (role === 'manufacturer') {
    // Look up companyId, add filter
    const user = await db.query.users.findFirst({
      where: eq(users.clerkId, getCurrentUser(req).userId!),
      columns: { companyId: true },
    });
    conditions.push(eq(brands.companyId, user!.companyId!));
  }
  // ... rest of existing logic
});
```

### Pattern 3: Notification Service Layer
**What:** A service module that creates notifications and sends emails in one call. Keeps route handlers clean.
**When to use:** Every approval workflow event
**Example:**
```typescript
// services/notification.ts
export async function notifyApprovalEvent(params: {
  type: 'listing_submitted' | 'listing_approved' | 'listing_rejected';
  listingId: number;
  listingName: string;
  recipientUserIds: number[];
  rejectionReason?: string;
}) {
  // 1. Insert notification rows for each recipient
  await db.insert(notifications).values(
    params.recipientUserIds.map(userId => ({
      userId,
      type: params.type,
      title: getTitleForType(params.type, params.listingName),
      message: getMessageForType(params.type, params.listingName, params.rejectionReason),
      entityType: 'listing',
      entityId: params.listingId,
    }))
  );

  // 2. Look up recipient emails and send via Resend
  const recipients = await db.query.users.findMany({
    where: inArray(users.id, params.recipientUserIds),
    columns: { email: true, firstName: true },
  });

  for (const recipient of recipients) {
    await sendApprovalEmail({
      to: recipient.email,
      type: params.type,
      listingName: params.listingName,
      rejectionReason: params.rejectionReason,
      ctaUrl: `${FRONTEND_URL}/manage/listings/${params.listingId}/edit`,
    });
  }
}
```

### Pattern 4: Listing Status State Machine
**What:** Enforce valid status transitions server-side.
**When to use:** All status-changing endpoints
**Valid transitions:**
```
draft -> pending_approval  (manufacturer submits)
pending_approval -> active (admin approves)
pending_approval -> rejected (admin rejects)
rejected -> pending_approval (manufacturer resubmits)
active -> archived (manufacturer archives)
archived -> pending_approval (manufacturer reactivates, requires re-approval)

Admin bypass: any -> any (admin can set any status directly)
```

### Anti-Patterns to Avoid
- **Client-side role filtering for security:** Never trust the frontend to hide data. ALL data scoping MUST happen in API WHERE clauses. The frontend role checks are UX only.
- **Separate manufacturer routes:** Don't create `/api/manufacturer/listings` -- use the same `/api/listings` with role-aware branching. Keeps code DRY and prevents drift.
- **Storing companyId in session token:** The JWT is managed by Clerk's session template. Adding companyId would require Clerk dashboard changes and couples auth to business data. Keep the DB lookup approach.
- **Notification fan-out at query time:** Don't query "all admins" every time a notification is read. Fan out at write time -- create one notification row per recipient user.

## Common Pitfalls

### Pitfall 1: Forgetting to scope brand dropdown for manufacturers
**What goes wrong:** Manufacturer sees all brands in the listing create form, can create listings under other companies' brands
**Why it happens:** The existing brand dropdown fetches all brands with no companyId filter
**How to avoid:** When role=manufacturer, pass companyId as query param to GET /api/brands so only company brands appear
**Warning signs:** Manufacturer can select brands they don't own in the listing form

### Pitfall 2: pgEnum migration requires ALTER TYPE
**What goes wrong:** Adding values to an existing Postgres enum requires `ALTER TYPE listing_status ADD VALUE 'pending_approval'`. Drizzle's `db:push` handles this, but generated migrations may need manual intervention.
**Why it happens:** Postgres enums are rigid; you can't remove values, only add them
**How to avoid:** Use `drizzle-kit push` for dev (handles ALTER TYPE automatically). For production, generate migration and verify the ALTER TYPE statement is correct.
**Warning signs:** Migration fails with "type already has value" or schema push doesn't detect enum changes

### Pitfall 3: Race condition on approval actions
**What goes wrong:** Two admins approve/reject the same listing simultaneously
**Why it happens:** No optimistic locking
**How to avoid:** Check current status before updating: `WHERE id = ? AND status = 'pending_approval'`. If zero rows affected, return 409 Conflict. This is sufficient for demo scale.
**Warning signs:** Listing ends up in unexpected state

### Pitfall 4: Notification email failures blocking approval actions
**What goes wrong:** Resend API is down or rate-limited, causing approval/rejection to fail
**Why it happens:** Email send is in the same request/response cycle as the DB update
**How to avoid:** Use try/catch around email sending. Log the error but don't fail the approval action. The in-app notification still gets created.
**Warning signs:** Approval buttons show errors even though the DB update succeeded

### Pitfall 5: Manufacturer editing listing while admin is reviewing
**What goes wrong:** Per user decisions, listings in pending_approval should be read-only for the manufacturer
**Why it happens:** PATCH endpoint doesn't check status before allowing edits
**How to avoid:** In the PATCH route for manufacturer role, check that listing status is NOT `pending_approval`. Return 403 with descriptive message. Also disable form controls on frontend.
**Warning signs:** Manufacturer can modify listing data while it shows as "pending" in the admin queue

### Pitfall 6: CompanyId null for admin users
**What goes wrong:** Admin user may not have a companyId (or may be in a different company type)
**Why it happens:** The seed data assigns admin to "AIS Admin Corp" (a manufacturer company), but the pattern should not assume admin always has companyId
**How to avoid:** Admin routes should never filter by companyId. Only manufacturer routes use companyId scoping. Always check role first.
**Warning signs:** Admin sees filtered data or gets errors when companyId is null

## Code Examples

### Notifications Table Schema
```typescript
// apps/api/src/db/schema/notifications.ts
import {
  boolean,
  integer,
  pgEnum,
  pgTable,
  text,
  timestamp,
  varchar,
} from 'drizzle-orm/pg-core';
import { users } from './users';

export const notificationTypeEnum = pgEnum('notification_type', [
  'listing_submitted',
  'listing_approved',
  'listing_rejected',
]);

export const notifications = pgTable('notifications', {
  id: integer().primaryKey().generatedAlwaysAsIdentity(),
  userId: integer('user_id')
    .notNull()
    .references(() => users.id),
  type: notificationTypeEnum().notNull(),
  title: varchar({ length: 255 }).notNull(),
  message: text().notNull(),
  entityType: varchar('entity_type', { length: 50 }),  // 'listing'
  entityId: integer('entity_id'),                       // listing.id
  isRead: boolean('is_read').notNull().default(false),
  createdAt: timestamp('created_at').defaultNow().notNull(),
});
```

**Key design decisions:**
- `userId` is the recipient (fan-out at write time, not read time)
- `entityType` + `entityId` enable "click to navigate" behavior
- No `updatedAt` needed since only `isRead` changes
- Types are limited to approval events now; extensible via ALTER TYPE later
- Index on `(userId, isRead)` for efficient unread count queries

### Resend Email Integration
```typescript
// apps/api/src/services/email.ts
import { Resend } from 'resend';

const resend = new Resend(process.env.RESEND_API_KEY);
const FROM_EMAIL = process.env.FROM_EMAIL || 'AIS Marketplace <noreply@ais-marketplace.com>';
const FRONTEND_URL = process.env.FRONTEND_URL || 'http://localhost:5173';

export async function sendApprovalEmail(params: {
  to: string;
  recipientName: string;
  type: 'listing_submitted' | 'listing_approved' | 'listing_rejected';
  listingName: string;
  rejectionReason?: string;
  listingId: number;
}) {
  const subject = {
    listing_submitted: `New listing submitted for review: ${params.listingName}`,
    listing_approved: `Your listing has been approved: ${params.listingName}`,
    listing_rejected: `Your listing needs changes: ${params.listingName}`,
  }[params.type];

  const ctaUrl = `${FRONTEND_URL}/manage/listings/${params.listingId}/edit`;
  const ctaLabel = {
    listing_submitted: 'Review Listing',
    listing_approved: 'View Listing',
    listing_rejected: 'Edit Listing',
  }[params.type];

  try {
    const { error } = await resend.emails.send({
      from: FROM_EMAIL,
      to: params.to,
      subject,
      html: buildEmailHtml({
        recipientName: params.recipientName,
        subject,
        bodyText: getEmailBody(params.type, params.listingName, params.rejectionReason),
        ctaUrl,
        ctaLabel,
      }),
    });
    if (error) {
      console.error('Resend email error:', error);
    }
  } catch (err) {
    console.error('Failed to send email:', err);
    // Don't throw -- email failure should not block approval actions
  }
}
```

## Open Questions

1. **Clerk session token and companyId**
   - What we know: The Clerk JWT contains `metadata.role` via custom session token template. companyId is NOT in the JWT. The `users` table has `companyId` and `clerkId`.
   - What's unclear: Whether to add companyId to the Clerk session template or keep the DB lookup approach
   - Recommendation: Keep the DB lookup approach. It is a single indexed query, avoids coupling auth to business data, and requires no Clerk dashboard changes. Create a `getCompanyUser()` helper or middleware that caches the lookup for the request lifecycle.

2. **Admin user targeting for notifications**
   - What we know: Notifications fan out at write time -- one row per recipient. When a manufacturer submits a listing, all admin users should receive a notification.
   - What's unclear: How many admin users will there be? Is "all admins" the right targeting?
   - Recommendation: Query `users WHERE role = 'admin'` at notification creation time. For demo scale this is fine. If admin count grows, consider a notification_subscriptions table later.

3. **Resend domain verification for dev/demo**
   - What we know: Resend requires domain verification for custom sender addresses. Free tier allows sending from `onboarding@resend.dev` for testing.
   - What's unclear: Whether AIS has a verified domain ready
   - Recommendation: Use `onboarding@resend.dev` for development. Add `RESEND_API_KEY` and `FROM_EMAIL` to root `.env`. Document domain verification as a deployment task.
