# Phase 2.1: Manufacturer Self-Service & Approval Workflow - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Manufacturers can create and manage listings and brands scoped to their company through the existing /manage/* pages. Listings go through an admin approval workflow (draft → pending_approval → active or rejected). Brands do NOT require approval. All users get a notification inbox with email notifications via Resend for approval events.

</domain>

<decisions>
## Implementation Decisions

### Manufacturer manage experience
- Reuse existing /manage/* routes (BrandsPage, ListingsPage) with role-based scoping — not separate /my/* routes
- Admin sees all data; manufacturer sees only their company's data (API filters by companyId from session)
- Same DataTable component for both roles; admin sees company column, manufacturer does not
- Manufacturer sidebar "Manage" item with sub-items: Brands, Listings (no Categories since read-only for manufacturers)
- Manufacturer can assign existing categories to listings but cannot create, edit, or delete categories
- Brand dropdown on listing create always shown (even if single brand) — no auto-assign logic

### Brand approval
- Brands created by manufacturers go active immediately — no approval gate for brands
- Admin can flag brand issues (name, logo, etc.) in listing rejection feedback if needed
- Simplification from original requirements: APRV-01 applies to listings only, not brands

### Listing approval workflow
- Extend existing listing_status enum with 'pending_approval' and 'rejected' (currently: draft, active, sold_out, archived)
- Manufacturer creates listing as 'draft' first — can edit freely in draft state
- Manufacturer explicitly clicks "Submit for Review" to move from draft → pending_approval
- Listing is read-only for manufacturer while in pending_approval (prevents admin reviewing a moving target)
- Admin approval moves listing directly to 'active' (no intermediate approved state)
- Admin rejection moves listing to 'rejected' with a rejection_reason stored on the listing (varchar column)
- Manufacturer can edit rejected listing and resubmit → clears rejection_reason, moves back to pending_approval
- Manufacturer can archive their own active listings; reactivating requires resubmission for approval
- SKUs inherit parent listing's approval status (approved/rejected as a unit)
- Admin-created listings bypass approval — go directly to active (or draft if admin chooses)

### Approval review queue
- Dedicated /manage/approvals page for admin (new sidebar sub-item under Manage)
- Single DataTable filtered to pending_approval listings only (no brand approvals)
- Inline action buttons per row: Approve, Reject, Edit
- Reject opens a small dialog for entering rejection reason
- Edit navigates to existing listing edit page
- Approve activates the listing immediately

### Rejection & status feedback
- Color-coded status badges in DataTable: draft (gray), pending_approval (yellow/amber), active (green), rejected (red), sold_out/archived (existing colors)
- Rejected listing shows inline alert banner with rejection reason at top of the edit page
- Rejection reason cleared on resubmit — no history tracking in Phase 2.1

### Notification inbox
- Full /notifications page — no dropdown panel
- Bell icon with unread count badge in the top header bar, beside the existing UserButton (Clerk) in the top-right
- Simple reverse-chronological list with "Mark all as read" button
- Each notification is manually marked as read (click/button per item)
- Clicking a notification navigates to the relevant entity (e.g., the listing) and marks it as read
- All user roles see the notification inbox

### Notification events (Phase 2.1 scope)
- Admin receives in-app notification when manufacturer submits listing for approval
- Manufacturer receives in-app notification when listing is approved or rejected (with reason)
- All approval events also send styled HTML email via Resend with AIS branding and CTA button linking to the relevant page
- Email notifications always on — no user toggle (NOTF-04 deferred to v2)

### Claude's Discretion
- Notification database schema design (table structure, indexes)
- Resend API integration details and email template markup
- Exact DataTable column selection for approvals page
- Loading states and error handling for approval actions
- How to extract companyId from Clerk session for API scoping
- Notification polling vs real-time strategy (polling is fine for Phase 2.1)

</decisions>

<specifics>
## Specific Ideas

- Approval queue is listings-only — brands don't need separate approval since they're just containers; admin can address brand issues in listing rejection feedback
- "Submit for Review" is an explicit action, not automatic on create — gives manufacturers time to polish listings before admin sees them
- Locked-while-pending prevents admin from reviewing a moving target
- Rejection reason stored on listing (not just in notification) so manufacturer always has context when editing
- Bell icon placement: next to Clerk UserButton in existing top header bar — no new header component needed

</specifics>
