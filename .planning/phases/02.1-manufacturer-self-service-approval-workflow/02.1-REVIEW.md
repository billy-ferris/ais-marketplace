# Phase 02.1: Code Review Report

**Reviewed:** 2026-06-30
**Depth:** standard
**Files Reviewed:** 33 (source + test) across apps/api, apps/web, packages/shared
**Status:** issues_found

## Summary

The approval state machine (draft → pending_approval → active/rejected, plus archive/revert-to-draft) is generally well built: the terminal transition endpoints (`submit`, `approve`, `reject`, `archive`, `revert-to-draft`) all use optimistic locking (status included in the `WHERE` clause) and correctly scope manufacturer access to their own company via `getCompanyUser`/`getListingForCompany`. Notification and email failures are correctly isolated (caught and logged, never block the approval action).

However, two real defects were found in adjacent code paths that were *not* covered by the same rigor as the dedicated approval endpoints:

1. **The general-purpose `PATCH /api/listings/:id` handler allows a manufacturer to reassign a listing's `brandId` to a brand it does not own**, because the ownership check only re-validates the *existing* brand, not the *new* one supplied in the request body. This is a genuine broken-access-control bug living right next to code that gets this right in `POST /api/listings`.
2. **User-controlled listing names and rejection reasons are interpolated unescaped into HTML email templates** (and the email subject line), creating an HTML/header-injection sink in `apps/api/src/services/email.ts`.

Several lower-severity robustness gaps were also found: unvalidated `status` query-param casts, a TOCTOU race in slug generation, a missing "exactly one primary image" invariant on the PATCH image-create path (present on POST but not PATCH), and an auth model that trusts Clerk session-claim `role` without any DB cross-check.

## Critical Issues

### CR-01: Manufacturer can reassign a listing's `brandId` to a brand belonging to another company (broken access control)

**File:** `apps/api/src/routes/listings.ts:420-465`

**Issue:** In `PATCH /:id`, the manufacturer ownership check only validates that the listing's *current* `brandId` belongs to the manufacturer's company:

```ts
// Verify listing belongs to manufacturer's company
const [brand] = await db
  .select({ companyId: brands.companyId })
  .from(brands)
  .where(eq(brands.id, existing.brandId));   // <-- existing.brandId, i.e. BEFORE update

if (!brand || brand.companyId !== user.companyId) {
  res.status(403).json({ error: 'Forbidden' });
  return;
}
```

`updateListingSchema` (derived from `createListingSchema.partial()`, see `packages/shared/src/schemas/listing.ts:11`) includes an optional `brandId` field. The subsequent update block does **not** strip or re-validate `brandId` for manufacturers — it only strips `status`:

```ts
const parsed = updateListingSchema.parse(listingData);
const updateValues = role === 'manufacturer'
  ? { ...parsed, status: undefined, updatedAt: new Date() }   // brandId passes through untouched
  : { ...parsed, updatedAt: new Date() };

await db.update(brandListings).set(updateValues).where(eq(brandListings.id, id));
```

A manufacturer can therefore send `PATCH /api/listings/:id` with `{ "listing": { "brandId": <any existing brand id> } }` and move their listing (and all of its SKUs/images/categories) onto a brand owned by a completely different company. This also leaks the target company's brand name/id in the subsequent re-query response.

Contrast with `POST /api/listings` (lines 275-295 of the same file), which correctly re-validates that `parsedListing.brandId` belongs to the manufacturer's company before insert — the PATCH path is simply missing the equivalent check for the *new* value.

**Fix:** Re-validate the target `brandId` for manufacturers when present in the payload, the same way `POST /` does:

```ts
if (role === 'manufacturer' && parsed.brandId !== undefined) {
  const [targetBrand] = await db
    .select({ companyId: brands.companyId })
    .from(brands)
    .where(and(eq(brands.id, parsed.brandId), isNull(brands.deletedAt)));
  if (!targetBrand || targetBrand.companyId !== user.companyId) {
    res.status(400).json({ error: 'Brand does not belong to your company' });
    return;
  }
}
```

### CR-02: Unescaped user input interpolated into approval email HTML and subject line (HTML/header injection)

**File:** `apps/api/src/services/email.ts:28-57, 80-90`

**Issue:** `listingName` (manufacturer-controlled, `brand_listings.name`, up to 255 chars with no character restrictions per `createListingSchema`) and `rejectionReason` (admin-controlled, up to 1000 chars) are interpolated directly, unescaped, into:

- the email **subject line**: `` `New listing submitted for review: ${name}` `` (line 29) — since `name` can contain any characters including CR/LF, this is a potential email-header-injection vector depending on how the Resend SDK serializes the subject.
- the email **HTML body**: `` `A new listing "<strong>${name}</strong>" has been submitted...` `` (lines 44-49) and again in `buildEmailHtml` (lines 80-85) — no HTML-escaping is applied, so a listing name like `<img src=x onerror=alert(1)>` or `</strong><a href="https://evil.example">Click here</a>` is rendered verbatim in the outbound HTML email to admins/manufacturers.

This is the classic unescaped-user-input-into-HTML-sink pattern; the reduced blast radius (recipients are known platform users, and most modern webmail clients strip `<script>`) does not eliminate the phishing/link-spoofing risk of injected markup, nor the header-injection risk in the subject.

**Fix:** Add a small HTML-escape helper and apply it to all interpolated user-controlled strings, and strip/reject control characters (`\r`, `\n`) from the subject-line inputs:

```ts
function escapeHtml(s: string): string {
  return s.replace(/&/g, '&amp;').replace(/</g, '&lt;').replace(/>/g, '&gt;')
          .replace(/"/g, '&quot;').replace(/'/g, '&#39;');
}
function sanitizeHeaderValue(s: string): string {
  return s.replace(/[\r\n]/g, ' ');
}
```
Apply `escapeHtml` in `BODY_TEXT_MAP` / `buildEmailHtml`, and `sanitizeHeaderValue` (plus `escapeHtml` is not needed there) before building `SUBJECT_MAP` output.

## Warnings

### WR-01: `findUniqueSlug` has a TOCTOU race that surfaces as an unhandled 500

**File:** `apps/api/src/routes/brands.ts:28-53`, `apps/api/src/routes/categories.ts:26-51`

**Issue:** `findUniqueSlug` does a `SELECT` to check for an existing slug, then the caller performs a separate `INSERT`. Two concurrent requests creating brands/categories with the same name can both pass the uniqueness check and then race to insert; the loser hits the DB-level `unique()` constraint (`brands.slug`, `categories.slug`) and throws, which is caught only by the generic `next(err)` → error-handler path (i.e., a raw 500 instead of a friendly 409/retry).

**Fix:** Either wrap the insert in a retry-on-unique-violation loop, or use `ON CONFLICT (slug) DO ...` semantics, or accept a Postgres error code `23505` in the route and re-run `findUniqueSlug`/retry once.

### WR-02: `PATCH /api/listings/:id` has no optimistic locking on status, unlike the dedicated transition endpoints

**File:** `apps/api/src/routes/listings.ts:405-465`

**Issue:** `submit`, `approve`, `reject`, `archive`, and `revert-to-draft` all include the expected status in the `UPDATE ... WHERE` clause (optimistic locking) to guard against concurrent transitions. The general `PATCH /:id` handler does not: it reads `existing.status` once (line 406-418) to decide whether the manufacturer is allowed to edit, then performs the update against `eq(brandListings.id, id)` only (line 464), with no re-check that the status hasn't changed in the interim. A narrow race exists where an admin's `approve`/`reject` completes between the manufacturer's status check and their field update, letting a stale edit slip through on a listing that has just become `active` (which should be immutable for manufacturers) or `pending_approval`.

**Fix:** Include `eq(brandListings.status, existing.status)` in the update `WHERE` clause for the manufacturer branch, and return 409 if zero rows affected (matching the pattern already used by the transition endpoints).

### WR-03: `images.create` on `PATCH /api/listings/:id` doesn't enforce the single-primary-image invariant, and silently drops images past the cap

**File:** `apps/api/src/routes/listings.ts:522-544`

**Issue:** `POST /api/listings` carefully computes `hasPrimary` across the whole batch and forces exactly one `isPrimary: true` (lines 334-343). The equivalent logic on `PATCH /:id` for `images.create` just does `isPrimary: !!img.isPrimary` (line 539) with no awareness of images that already exist on the listing. Two problems result:
1. If a client sends a new image with `isPrimary: true` while an existing image is already primary, the listing ends up with two `isPrimary: true` images (no invariant enforcement, no DB constraint).
2. If `remaining <= 0` (already at the 5-image cap), the new images are silently dropped (line 531 `if (remaining > 0)`) with no error/warning returned to the client — the caller has no way to know their images were not saved.

**Fix:** Before inserting, unset `isPrimary` on existing images if a new image claims it (or reject the request if more than one `isPrimary: true` is present), and return a 400 (or a partial-success indicator) when `images.create.length > remaining`.

### WR-04: `status` query param on `GET /api/listings` is cast, not validated

**File:** `apps/api/src/routes/listings.ts:90-92`

**Issue:**
```ts
if (status) {
  conditions.push(eq(brandListings.status, status as 'draft' | 'active' | 'sold_out' | 'archived' | 'pending_approval' | 'rejected'));
}
```
The query parameter is blindly type-asserted to the enum type with no runtime check. An invalid value (`?status=bogus`) is passed straight to Postgres, which will raise `invalid input value for enum listing_status`, propagating as an unhandled 500 via `next(err)` rather than a clean 400.

**Fix:** Validate against `listingStatusEnum.enumValues` (or the shared `ListingStatus` object) before adding the condition, returning 400 on mismatch.

### WR-05: Rejection reason has no max-length validation matching the DB column

**File:** `apps/api/src/routes/listings.ts:825-829`

**Issue:** `brand_listings.rejection_reason` is `varchar(1000)` (`apps/api/src/db/schema/brand-listings.ts:29`), but the reject handler only checks that `reason` is a non-empty string — there is no upper bound:
```ts
const { reason } = req.body;
if (!reason || typeof reason !== 'string' || reason.trim().length === 0) {
  res.status(400).json({ error: 'Rejection reason is required' });
  return;
}
```
A reason longer than 1000 characters will pass this check and then fail at the DB layer with an unhandled "value too long for type character varying(1000)" error (500), rather than a clean validation 400.

**Fix:** Add `reason.trim().length > 1000` to the validation block (or move validation into a Zod schema shared with the column length).

### WR-06: Role-based authorization trusts Clerk session-claim `role` with no DB cross-check

**File:** `apps/api/src/middleware/auth.ts:31-43`

**Issue:** `requireRole` reads `auth.sessionClaims?.metadata?.role` from the Clerk JWT only — it never cross-checks against the local `users.role` column. Company-scoping (`getCompanyUser`) does hit the DB fresh on every request, but the coarse admin/manufacturer/retailer gate does not. If a user's role is changed in the local DB (or via an admin action) faster than their Clerk session token/metadata is refreshed, the stale JWT will continue to grant the old role's permissions for the life of the session token. This is a common trade-off with JWT-based claims, but given this phase adds admin-only approval actions (`approve`/`reject`), it's worth flagging as a deliberate risk to confirm is acceptable, or to mitigate with shorter session token TTLs / webhook-driven forced re-auth on role change.

**Fix (optional/defense-in-depth):** For high-privilege actions (e.g., `approve`/`reject`), consider an additional DB-role check via `getCompanyUser`/a dedicated `getLocalUser` lookup rather than relying solely on session claims.

### WR-07: Manufacturer-supplied `categoryIds` on `PATCH /api/listings/:id` aren't validated to exist before insert

**File:** `apps/api/src/routes/listings.ts:562-577`

**Issue:** The category-replace block deletes all existing `listing_categories` rows for the listing and blindly re-inserts whatever `categoryIds` were supplied, with no check that each ID references an existing (non-deleted) category:
```ts
if (categoryIds.length > 0) {
  await db.insert(listingCategories).values(
    categoryIds.map((categoryId: number) => ({ listingId: id, categoryId })),
  );
}
```
An invalid `categoryId` throws an FK-constraint error (unhandled 500) instead of a clean 400, and — because the existing rows were already deleted before the insert is attempted — a failed insert leaves the listing with **zero** categories instead of rolling back to its prior state (no transaction wraps the delete+insert pair).

**Fix:** Validate `categoryIds` against `categories` (existing, non-deleted) before the delete, and/or wrap the delete+insert in a DB transaction so a bad ID doesn't leave the listing category-less.

## Info

### IN-01: `getCompanyUser` contains dead/misleading `?? null`

**File:** `apps/api/src/middleware/auth.ts:63-77`

**Issue:**
```ts
export async function getCompanyUser(req: Request) {
  const { userId: clerkId } = getCurrentUser(req);
  if (!clerkId) return null;

  return db.query.users.findFirst({ ... }) ?? null;
}
```
`db.query.users.findFirst(...)` returns a `Promise`, and `?? null` is applied to that (always-truthy) `Promise` object, not to its eventually-resolved value — so this line never actually converts a not-found result to `null`. Because the function is `async`, returning a promise still works correctly (the outer promise adopts the inner one's resolution), so callers see `undefined` (not `null`) when no user is found. This doesn't currently break anything because every call site uses optional chaining (`user?.companyId`), but the `?? null` reads as intentional null-normalization and isn't — worth removing or fixing (`return (await db.query...) ?? null;`) to avoid future confusion.

**Fix:** `return (await db.query.users.findFirst({...})) ?? null;`

### IN-02: `ListingForm.tsx` bridges create/update SKU & image payload shapes with unsafe type casts

**File:** `apps/web/src/components/manage/ListingForm.tsx:294-325`

**Issue:** The submit handler builds `payload.skus`/`payload.images` as the PATCH-shaped `{ create, update, delete }` object, then for create-mode overwrites them with flat arrays cast via `as unknown as ReturnType<typeof buildSkuPayload>` (line 317-319) and `as unknown as typeof payload.images` (line 323) to match the different shape `POST /api/listings` expects. This works today because both ends of the contract happen to agree, but the `as unknown as` casts fully suppress type-checking — if the backend's POST/PATCH payload shapes ever diverge further, TypeScript will not catch it.

**Fix:** Model the two payload shapes as a discriminated union or separate types so the compiler can verify both branches, rather than casting through `unknown`.

### IN-03: `ListingForm.tsx` status `Select` keys off the display label instead of the stable enum value

**File:** `apps/web/src/components/manage/ListingForm.tsx:399-419`

**Issue:** Both `BrandDialog.tsx` and the brand `Select` in `ListingForm.tsx` explicitly comment that they use the stable numeric ID as the `Select` value "so [options] that share a display name can't collide." The status `Select` in the same file does the opposite — it uses `LISTING_STATUS_LABELS[selectedStatus]` (the label string) as the `value`, and resolves back to the enum via `STATUS_OPTIONS.find((o) => o.label === label)` (line 403). This is inconsistent with the pattern the code otherwise follows and is fragile if two status labels are ever made to match (or are localized).

**Fix:** Use `opt.value` (the enum key) directly as the `Select` value, matching the pattern used for brand/company selects elsewhere in the same file/component.

### IN-04: Approval-workflow ownership/role checks are duplicated across five endpoints

**File:** `apps/api/src/routes/listings.ts` (submit: 661-741, approve: 744-814, reject: 817-894, archive: 897-964, revert-to-draft: 967-1034)

**Issue:** The pattern "look up `existing`, if manufacturer fetch `getCompanyUser`/`getListingForCompany`, check status, apply optimistic-locked update" is repeated near-verbatim across five route handlers. This isn't a functional bug (each copy is currently correct, aside from CR-01/WR-02 above), but the duplication increases the chance that a future edit to one copy (e.g., fixing WR-02) is not propagated to the others.

**Fix:** Extract a shared helper (e.g., `transitionListingStatus(id, role, companyId, { from, to, extraChecks })`) to centralize the ownership + status-transition logic.
