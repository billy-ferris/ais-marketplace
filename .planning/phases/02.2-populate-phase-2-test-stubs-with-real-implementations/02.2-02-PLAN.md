3. Mock db module with chainable methods. All chain methods return `this` via mockReturnThis(). Terminal methods (returning, the final chainable that resolves) use mockResolvedValue(). The db mock must handle Promise.all for list routes (data query + count query).

```typescript
const mockDb: any = {};
vi.mock('../../db/index', () => {
  // Build chain mock -- each method returns mockDb for chaining
  mockDb.select = vi.fn().mockReturnValue(mockDb);
  mockDb.from = vi.fn().mockReturnValue(mockDb);
  mockDb.where = vi.fn().mockReturnValue(mockDb);
  mockDb.leftJoin = vi.fn().mockReturnValue(mockDb);
  mockDb.limit = vi.fn().mockReturnValue(mockDb);
  mockDb.offset = vi.fn().mockReturnValue(mockDb);
  mockDb.orderBy = vi.fn().mockReturnValue(mockDb);
  mockDb.insert = vi.fn().mockReturnValue(mockDb);
  mockDb.values = vi.fn().mockReturnValue(mockDb);
  mockDb.returning = vi.fn().mockResolvedValue([]);
  mockDb.update = vi.fn().mockReturnValue(mockDb);
  mockDb.set = vi.fn().mockReturnValue(mockDb);
  mockDb.delete = vi.fn().mockReturnValue(mockDb);
  // Make mockDb itself thenable for Promise.all resolution
  mockDb.then = undefined; // Will be set per test
  return { db: mockDb };
});
```

IMPORTANT: The brand routes use `Promise.all([dataQuery, countQuery])`. Both queries go through the same mockDb chain. The simplest approach: make mockDb awaitable by setting a custom `.then` or by overriding the terminal chain result per test.

Alternative simpler approach: Since the mock chain returns mockDb, and Promise.all awaits both entries, override `mockDb[Symbol.for('thenResult')]` or use a different technique -- make the final chained call (`orderBy` for data, `where` for count) resolve to the expected data.

Recommended approach for list routes: Reset mock behavior in beforeEach. For list route tests, configure mockDb so that when the chain is awaited in Promise.all, the first resolution gives data array and the second gives [{total: N}]. The cleanest way:
- Make mockDb implement a `.then()` method that resolves to the configured return data, then sets itself up for the next call.
- Or, since both queries share the chain, use mockReturnValueOnce pattern on a terminal method.

Actually, the simplest working approach: Since Promise.all([query1, query2]) awaits both, and both go through mockDb chains that end differently (query1 ends with .orderBy(), query2 ends with .where()), we can make the mock resolve at the point where it becomes a Promise. The Drizzle query builder is thenable. Make mockDb thenable:

```typescript
// In beforeEach for list tests:
let callCount = 0;
mockDb.then = vi.fn((resolve: any) => {
  callCount++;
  if (callCount === 1) resolve(mockBrandData); // data query
  else resolve([{ total: 1 }]); // count query
});
```

For non-list tests (GET /:id, POST, PATCH, DELETE), disable mockDb.then and use .returning() mock.

function getHandler(router: any, method: string, path: string) {
  const layer = router.stack.find((l: any) =>
    l.route?.path === path && l.route?.methods[method]
  );

  return layer?.route?.stack[layer.route.stack.length - 1]?.handle;
}

**CRITICAL mock management:** In beforeEach, call vi.clearAllMocks() and reset mockDb.then to undefined (non-thenable by default). Each test configures the mock behavior it needs. For list tests, set mockDb.then. For single-item tests, make the chain resolve via then. For insert/update/delete tests, configure mockDb.returning.

The challenge is that all db chain methods return mockDb (same object). For the list GET route which uses Promise.all on two separate chains, both chains resolve through mockDb.then. Use a call counter in then() to return different data for each call.

For GET /:id, POST returning, PATCH returning, DELETE returning -- these await the chain directly (not via Promise.all). The chain ends with .returning() for mutation routes, or the chain itself is awaited for select queries.

If mockDb.then proves fragile, an alternative that works: make the chain terminal methods resolve differently.

ACTUALLY, the most reliable approach per the research: for routes that use `const [brand] = await db.select().from().where()`, the chain is awaited directly. Since all methods return mockDb, mockDb itself is what gets awaited. If mockDb has a `.then` method, it becomes a thenable. This is the correct approach. Set mockDb.then in each test to control what the awaited chain resolves to. Reset in beforeEach.

For routes using `.returning()`, the returning() mock's resolvedValue is what the await gets. So those tests configure mockDb.returning.mockResolvedValue([expected]).

For POST, which calls findUniqueSlug (db.select chain) THEN db.insert().values().returning(), you need both: mockDb.then for the select chain in findUniqueSlug, and mockDb.returning for the insert.
