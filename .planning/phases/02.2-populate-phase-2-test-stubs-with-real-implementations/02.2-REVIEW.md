# Phase 02.2: Code Review Report

**Reviewed:** 2026-07-01T02:45:48Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

`catalog.test.ts` (Zod schema tests) is solid — assertions map directly to schema constraints defined in `packages/shared/src/schemas/*.ts` and would fail if those constraints changed. The route test files (`brands.test.ts`, `categories.test.ts`, `listings.test.ts`, `uploads.test.ts`) are structurally sound (correct `vi.hoisted`/`vi.mock` ordering, proper thenable-mock resets for `Promise.all` chains) but contain a recurring pattern of **tautological assertions**: many tests titled "should filter/exclude X" only assert that `mockDb.where()` or `res.json()` was called — which is true on every request regardless of whether the filter condition was actually applied — rather than asserting on the condition/arguments passed to the query builder. These tests would pass unchanged even if the underlying filter logic in the route were deleted. I also found a genuine test-isolation bug (auth-role mock state leaking across tests because `vi.clearAllMocks()` doesn't reset `mockReturnValue`), a weak assertion that masks a real double-`isPrimary` bug in `listings.ts`, and a whole endpoint (`POST /:id/revert-to-draft`) with zero test coverage. `seed/categories.test.ts` never actually exercises `apps/api/src/db/seed.ts` despite its `describe('seed script', ...)` / test titles implying it does — it only checks static DB-schema/FK metadata.

## Warnings

### WR-01: Tautological "exclude soft-deleted" tests

**File:** `apps/api/src/__tests__/routes/brands.test.ts:194-212`, `apps/api/src/__tests__/routes/categories.test.ts:179-197`, `apps/api/src/__tests__/routes/listings.test.ts:248-256`
**Issue:** Each of these tests is titled "should exclude soft-deleted X" but only asserts `expect(mockDb.where).toHaveBeenCalled()` and that `res.json` returns the mock's canned data. `where()` is called unconditionally on every list query (it always includes at least `isNull(...deletedAt)`), so this assertion is true whether or not the soft-delete filter exists in the route. If the `isNull(brands.deletedAt)` (or equivalent) condition were deleted from `brands.ts`/`categories.ts`/`listings.ts`, these tests would still pass.
**Fix:** Capture the actual condition passed to `.where()` and assert on it, e.g.:
```ts
expect(mockDb.where).toHaveBeenCalledWith(
  expect.objectContaining({ /* match the drizzle SQL/condition shape, or */ }),
);
// or, more robustly, spy on `isNull`/`and` from drizzle-orm and assert
// they were invoked with `brands.deletedAt` / `categories.deletedAt`.
```
At minimum, add a companion test where the mock DB is configured to return a row that would only appear if the filter were missing (e.g. simulate a deleted-but-returned row) and assert it is NOT present in the exposed response — right now every "exclude" test only ever feeds already-filtered data into the mock.

### WR-02: "Filter by X" list-query tests don't verify the filter condition

**File:** `apps/api/src/__tests__/routes/brands.test.ts:156-192` (search, companyId), `apps/api/src/__tests__/routes/categories.test.ts:142-159` (search), `apps/api/src/__tests__/routes/listings.test.ts:189-218` (search, status, brandId), `apps/api/src/__tests__/routes/listings.test.ts:736-758` (manufacturer scoping)
**Issue:** Tests such as "should filter brands by search query (ilike on name)" and "should filter by status" only assert `mockDb.where` was called and `res.json` was called — never that `ilike(brands.name, '%test%')`, `eq(brandListings.status, 'active')`, `eq(brands.companyId, ...)`, etc. were actually built. Since `.where()` fires on every request path (filtered or not), removing the entire search/status/brandId/company-scoping branch from the route would not break any of these tests.
**Fix:** Assert on the arguments the handler passes downstream, e.g. capture `req.query` handling via a spy on `ilike`/`eq` from `drizzle-orm`, or (simpler) assert on the shape of data returned for a scenario that only makes sense if filtering happened (e.g. seed the mock with both a matching and a non-matching row and assert only the matching row appears — currently the mocks are pre-filtered by the test itself, so nothing exercises the route's own filtering).

### WR-03: Auth-role mock state leaks across tests (test isolation bug)

**File:** `apps/api/src/__tests__/routes/brands.test.ts:434-577`, `apps/api/src/__tests__/routes/listings.test.ts:736-1146`
**Issue:** `getCurrentUser`/`getCompanyUser` are mocked via `vi.fn().mockReturnValue(...)` at module scope, defaulting to `role: 'admin'`. Several tests call `(getCurrentUser as any).mockReturnValue({ role: 'manufacturer' })` to test manufacturer-scoped behavior. `beforeEach` only calls `vi.clearAllMocks()`, which clears call history but **does not** reset a mock's `mockReturnValue` (that requires `mockReset()`/`resetAllMocks()`, confirmed absent from both `vitest.config.ts` and `src/__tests__/setup.ts`). As a result, once a test sets the role to `'manufacturer'`, every subsequent test in the file silently inherits `role: 'manufacturer'` from `getCurrentUser` unless it explicitly re-mocks back to `'admin'`. Several tests do this defensively (e.g. `listings.test.ts:797-799`, `:1062-1063`, `:1097-1098`), which the authors clearly know is needed — but it's manual and easy to forget. Any new test added after a manufacturer-role test, or any reordering (e.g. `vitest --sequence.shuffle`), will silently run under the wrong role and either produce false failures or, worse, false passes if the handler path happens not to diverge by role.
**Fix:** Reset auth mocks in `beforeEach` alongside the `mockDb` reset:
```ts
beforeEach(() => {
  vi.clearAllMocks();
  (getCurrentUser as any).mockReturnValue({ userId: 'test-user', role: 'admin', sessionClaims: { metadata: { role: 'admin' } } });
  (getCompanyUser as any).mockResolvedValue({ id: 1, companyId: 1, role: 'admin', email: 'admin@test.com', firstName: 'Admin' });
  // ...existing mockDb resets
});
```

### WR-04: Weak assertion masks a real "multiple primary images" bug

**File:** `apps/api/src/__tests__/routes/listings.test.ts:408-453`
**Issue:** The test "should set exactly one image as primary" asserts `expect(primaryCount).toBeGreaterThanOrEqual(1)` instead of `toBe(1)`. Worse, the test never submits more than one image with `isPrimary: true` explicitly. Looking at the implementation (`apps/api/src/routes/listings.ts:335-343`):
```ts
const hasPrimary = imageSlice.some((img) => img.isPrimary);
const imageValues = imageSlice.map((img, index) => ({
  ...
  isPrimary: hasPrimary ? !!img.isPrimary : index === 0,
}));
```
when `hasPrimary` is true (i.e. the caller explicitly flagged ≥1 image as primary), **every** image keeps its own `!!img.isPrimary` value — so a caller submitting two images both with `isPrimary: true` would insert two primary images, silently violating "exactly one primary" invariant. The current test cannot catch this because (a) it never submits multiple `isPrimary: true` images, and (b) even if it did, `toBeGreaterThanOrEqual(1)` would still pass.
**Fix:**
```ts
it('should set exactly one image as primary', async () => {
  // ...
  expect(primaryCount).toBe(1); // not >= 1
});

it('should coerce multiple explicit isPrimary flags down to exactly one', async () => {
  const images = [
    { imageUrl: 'https://img.test/0.jpg', displayOrder: 0, isPrimary: true },
    { imageUrl: 'https://img.test/1.jpg', displayOrder: 1, isPrimary: true },
  ];
  // ... assert only one inserted image has isPrimary === true
});
```
This will currently fail against the real implementation and should be filed as a bug against `listings.ts`, not just fixed in the test.

### WR-05: GET /:id soft-delete filtering never tested with a negative case

**File:** `apps/api/src/__tests__/routes/listings.test.ts:263-292` ("should return full listing..."), `apps/api/src/__tests__/routes/listings.test.ts:304-323` ("should exclude soft-deleted SKUs from nested response")
**Issue:** Both tests only ever feed `deletedAt: null` rows into the mocked `db.query.brandListings.findFirst`. The comment on the SKU test even admits this: "The query itself filters deleted SKUs via the where clause in with. The mock returns only non-deleted SKUs (as the real DB would)." Because `db.query.*` is mocked wholesale, the test never verifies that `where: isNull(inventorySkus.deletedAt)` is actually part of the `with.inventorySkus` config, nor that `.filter((lc) => lc.category.deletedAt === null)` in the handler actually strips a deleted category. If either filter were removed from `listings.ts`, no test in this file would notice.
**Fix:** Add a case where the mocked `findFirst` return includes a deleted SKU/category (`deletedAt: new Date()`) alongside a live one, and assert the handler still returns only the live one where the handler does its own filtering (categories), or assert on the exact `with` config object passed to `findFirst` for the SKU case (since that filter lives in the query, not the handler):
```ts
expect(mockDb.query.brandListings.findFirst).toHaveBeenCalledWith(
  expect.objectContaining({
    with: expect.objectContaining({
      inventorySkus: expect.objectContaining({ where: expect.anything() }),
    }),
  }),
);
```

### WR-06: No test for filename sanitization / path-traversal in uploads route

**File:** `apps/api/src/__tests__/routes/uploads.test.ts` (entire `POST /presigned-url` suite, lines 85-172)
**Issue:** `sanitizeFileName()` in `apps/api/src/routes/uploads.ts:21-26` strips characters outside `[a-zA-Z0-9.\-_]` but leaves `.` untouched, so a `fileName` like `../../etc/passwd.jpg` becomes `..-..-etc-passwd.jpg` (slashes become hyphens, `..` sequences are preserved). No test in `uploads.test.ts` exercises a `fileName` containing `/`, `..`, or other traversal/injection-relevant characters to confirm the resulting S3 key is safe.
**Fix:**
```ts
it('should strip slashes and traversal sequences from fileName', async () => {
  const { req, res, next } = createMockReqRes({
    body: { fileName: '../../etc/passwd.jpg', contentType: 'image/jpeg' },
  });
  await handler(req, res, next);
  const payload = (res.json as any).mock.calls[0][0];
  expect(payload.key).not.toMatch(/\.\.|\//.test ? /\.\.\//: /\.\.\//);
  expect(payload.key.split('/')).toHaveLength(2); // "images", "<sanitized>"
});
```

### WR-07: "should require admin role" tests don't verify the actual role

**File:** `apps/api/src/__tests__/routes/brands.test.ts:320-327`, `apps/api/src/__tests__/routes/uploads.test.ts:131-141`
**Issue:** Both tests only assert `layer.route.stack.length > 1` (or `>= 3`). This proves *some* middleware chain exists, not that `requireRole('admin')` specifically was registered. If the route were accidentally changed to `requireRole('manufacturer')` or `requireRole('admin', 'manufacturer')`, both tests would still pass unchanged.
**Fix:** Assert on the mocked `requireRole` call arguments instead of stack length, e.g. capture the factory call during module import:
```ts
// at top of file, before importing the router:
import { requireRole } from '../../middleware/auth';
// ...
it('should require admin role', () => {
  expect(requireRole).toHaveBeenCalledWith('admin');
});
```
(Note: `requireRole` is already imported unused in both files — see IN-01.)

### WR-08: Misleading test name in listings.test.ts

**File:** `apps/api/src/__tests__/routes/listings.test.ts:760-794`
**Issue:** The test is titled "should filter brand dropdown to manufacturer company brands" but the test body posts to `POST /` (create listing) and asserts brand-ownership validation during listing creation — there is no "brand dropdown" endpoint or filtering behavior exercised anywhere in this test. The name doesn't describe what's actually being verified, making the suite harder to maintain and potentially hiding that brand-dropdown-filtering functionality (if it exists elsewhere) has no coverage at all.

### WR-09: seed/categories.test.ts never exercises the actual seed script

**File:** `apps/api/src/__tests__/seed/categories.test.ts:47-112`
**Issue:** The `describe('seed script', ...)` block and its test titles ("should seed all 7 categories into the database", "should seed brands linked to manufacturer companies", "should seed listings with SKUs") strongly imply the seed logic in `apps/api/src/db/seed.ts` is under test. In reality, `seed.ts` calls `main()` as a top-level side effect on import (connects to a real Postgres DB and Clerk), so it cannot be imported directly in a unit test — the test file works around this by only checking static Drizzle schema metadata (`getTableColumns`, `getTableConfig`, FK targets) and the shape of the `CPG_CATEGORIES` constant. None of the actual seeding logic in `seed.ts` is exercised: the `seedListings`/`seedSkusByListing` index alignment (`seedSkusByListing[i]` must correspond to `seedListings[i]`), the `categorySlugs` → `categoryBySlug` lookup, or the brand/company index mapping. An off-by-one or typo'd slug in `seed.ts` (e.g. `categorySlugs: ['hair-car']`) would produce silently-dropped category assignments (the code does `if (categoryId) { ... }` — no error on missing match) and no test would catch it.
**Fix:** Either (a) refactor `seed.ts` to export the pure data-transformation pieces (e.g. `buildCategoryValues(CPG_CATEGORIES)`, `resolveCategoryAssignments(seedListings, categoryBySlug)`) so they can be unit tested without touching a real DB/Clerk, and add tests asserting every `categorySlugs` entry across `seedListings` resolves to a valid slug in `CPG_CATEGORIES`; or (b) rename the `describe`/tests to accurately reflect that they test *schema shape*, not the seed script (e.g. `describe('Catalog schema supports seed data shape', ...)`), so the coverage gap isn't hidden.

### WR-10: Zero test coverage for `POST /:id/revert-to-draft`

**File:** `apps/api/src/__tests__/routes/listings.test.ts` (missing — compare to `apps/api/src/routes/listings.ts:967-1034`)
**Issue:** `listings.ts` defines five approval-workflow endpoints (`/:id/submit`, `/:id/approve`, `/:id/reject`, `/:id/archive`, `/:id/revert-to-draft`). The test file's "Approval workflow" describe block covers submit, approve, reject, archive — but `revert-to-draft` has no test at all (not even a happy-path test), despite having the same optimistic-locking/ownership-check shape as `archive` right above it in the source.
**Fix:** Add at minimum a happy-path test (`archived -> draft` for both admin and manufacturer) and a conflict test (`existing.status !== 'archived'` returns 400), mirroring the existing archive tests at `listings.test.ts:1115-1145`.

### WR-11: Missing test for reject-without-reason validation branch

**File:** `apps/api/src/__tests__/routes/listings.test.ts` (missing — compare to `apps/api/src/routes/listings.ts:825-829`)
**Issue:** `POST /:id/reject` returns `400 { error: 'Rejection reason is required' }` when `reason` is missing, not a string, or blank/whitespace-only. Only the success path (`reason: 'Quality issues'`) is tested at `listings.test.ts:988-1023`. This validation branch — which prevents admins from rejecting a listing without an audit trail — has zero coverage.
**Fix:**
```ts
it('should reject with 400 when reason is missing or blank', async () => {
  const handler = getHandler(listingRouter, 'post', '/:id/reject');
  for (const body of [{}, { reason: '' }, { reason: '   ' }]) {
    const { req, res, next } = createMockReqRes({ params: { id: '1' }, body });
    await handler(req, res, next);
    expect(res.status).toHaveBeenCalledWith(400);
    expect(res.json).toHaveBeenCalledWith({ error: 'Rejection reason is required' });
  }
});
```
