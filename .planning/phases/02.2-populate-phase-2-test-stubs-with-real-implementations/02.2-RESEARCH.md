## Summary

This phase fills out 85 `it.todo()` test stubs across 6 existing test files with real test implementations. The stubs were created during Phase 2 as Wave 0 infrastructure. The test files cover three distinct testing domains: (1) Zod schema validation (21 stubs -- pure unit tests, no mocking needed), (2) Drizzle ORM schema introspection (7 stubs + 7 existing passing tests as pattern reference), (3) Express route handler tests (57 stubs -- require mocking db and auth), and (4) seed data constants validation (7 stubs -- pure assertion tests).

The project already has two fully-implemented test files (`rbac.test.ts` with 5 passing tests and `schema.test.ts` with 8 passing tests) that establish clear patterns. The key decision for route tests is whether to test at the HTTP level (supertest) or at the handler/mock level. Since supertest is not installed and the project uses direct `vi.mock()` for `@clerk/express` in `rbac.test.ts`, the established pattern is **handler-level unit tests with mocked dependencies** -- not HTTP integration tests.

**Primary recommendation:** Implement all 85 stubs as pure unit tests using `vi.mock()` for `db`, `@clerk/express`, and `@aws-sdk`. Follow the existing `rbac.test.ts` mock pattern for auth and the `schema.test.ts` `getTableColumns`/`getTableConfig` pattern for schema tests. Zod schema tests need zero mocking. Route tests mock the `db` module and call route handlers directly via mock `req`/`res` objects.

### Pattern 1: Mock Express req/res for Route Handler Tests
**What:** Create mock request/response objects and call middleware/handler functions directly
**When to use:** All route handler tests (brands, categories, listings, uploads)
**Example (from rbac.test.ts):**
```typescript
import { describe, it, expect, vi, beforeEach } from 'vitest';
import type { Request, Response, NextFunction } from 'express';

// Mock external dependencies before imports
vi.mock('@clerk/express', () => ({
  clerkMiddleware: vi.fn(),
  requireAuth: vi.fn(() => (_req: Request, _res: Response, next: NextFunction) => next()),
  getAuth: vi.fn(),
}));

function createMockReqRes(overrides?: { params?: Record<string, string>; query?: Record<string, string>; body?: unknown }) {
  const req = {
    params: overrides?.params ?? {},
    query: overrides?.query ?? {},
    body: overrides?.body ?? {},
  } as unknown as Request;
  const res = {
    status: vi.fn().mockReturnThis(),
    json: vi.fn().mockReturnThis(),
  } as unknown as Response;
  const next = vi.fn() as NextFunction;
  return { req, res, next };
}
```

### Anti-Patterns to Avoid
- **Testing implementation details:** Do NOT assert on the exact SQL queries Drizzle generates. Test the behavior (what the handler returns) not the mechanics (which Drizzle method chain was used).
- **Over-mocking Zod:** Zod schemas are pure functions. Do NOT mock Zod -- just call `.safeParse()` and assert on the result.
- **Integration tests without a real DB:** The route test stubs describe unit-test-level assertions. Do NOT try to spin up a test database for these tests. The mocked db pattern is appropriate.
- **Testing Clerk auth in route tests:** Auth middleware is already tested in `rbac.test.ts`. Route tests should mock auth as pass-through (or mock it to return the expected role). Don't re-test auth behavior in every route test.

## Common Pitfalls

### Pitfall 1: Drizzle Chain Mocking Complexity
**What goes wrong:** Drizzle uses a fluent builder pattern (`db.select().from().where().limit()...`). Each method must return `this` (the mock) to allow chaining, and the terminal method (`.returning()`, the final `.from()`, etc.) must return the actual data.
**Why it happens:** The db mock needs to handle chains of arbitrary length.
**How to avoid:** Use `mockReturnThis()` for all chainable methods and `mockResolvedValue()` on terminal methods. Reset the terminal method's return value in `beforeEach`.
**Warning signs:** Tests fail with "Cannot read property 'from' of undefined" -- means a chain link is missing `.mockReturnThis()`.

### Pitfall 2: Promise.all in List Routes
**What goes wrong:** The brands/categories/listings GET routes use `Promise.all([dataQuery, countQuery])`. Both queries need to resolve for the route handler to proceed.
**Why it happens:** The mock db is shared between the data query and the count query, so the chain mock return value for the terminal method gets called twice.
**How to avoid:** For list routes, mock the db chain to return different values on successive calls (first for data, then for count), or restructure the mock to handle both query paths. The simplest approach: have the mock chain's terminal resolve to `[data, [{ total: N }]]` when the route calls `Promise.all`.
**Warning signs:** Tests hang or return `undefined` for pagination data.

### Pitfall 3: Route Handler Extraction
**What goes wrong:** The route handlers are defined inline on `router.get()`, `router.post()`, etc. There is no exported handler function to call directly.
**Why it happens:** Express routes are typically registered on a Router instance, not exported as standalone functions.
**How to avoid:** Two approaches: (a) Import the router and use the Express Router's internal stack to extract handlers, or (b) mock the entire middleware chain including auth and call the handler in sequence. Approach (b) is simpler and more maintainable. The route files export the router, so you can mock `requireAuth` and `requireRole` to be pass-through, then mount the router in a mini Express app and make requests using the mock req/res pattern OR by directly invoking the handler chain from the router's stack.
**Recommended approach:** Extract handlers from the router's internal layer stack. Express Router stores route handlers as layers. Access via `router.stack` to find the matching path and method, then call the handler directly.

### Pitfall 4: ZodError Propagation in Route Tests
**What goes wrong:** Route handlers call `createBrandSchema.parse(req.body)` which throws `ZodError` on invalid input. The handler catches this in `catch(err) { next(err) }` which passes to the error handler middleware.
**Why it happens:** The error handler is not in the route handler scope -- it is registered on the Express app.
**How to avoid:** For "should reject invalid data" tests, either: (a) assert that `next` was called with a `ZodError` instance, or (b) use the actual error handler. Approach (a) is simpler for unit tests.
**Warning signs:** Tests expect `res.status(400)` but it never fires because the error goes to `next()`.

### Pitfall 5: Seed Script Tests That Need a Database
**What goes wrong:** The seed/categories.test.ts has 3 stubs about "seed script" behavior (seeding categories into DB, brands linked to companies, listings with SKUs). These describe integration test behavior.
**Why it happens:** The stubs were written to describe end-to-end seed behavior, but the seed script requires a real database and Clerk API.
**How to avoid:** For the 3 "seed script" stubs, test the seed DATA DEFINITIONS (the `seedBrands`, `seedListings`, `seedSkusByListing` arrays) as structural assertions rather than actually running the seed. Alternatively, test that the seed script's data definitions are consistent (e.g., every listing's brandIndex maps to a valid brand, every SKU has required fields). If the seed data is not exported, the tests can import the seed module with a mocked db and verify the insert calls.

## Route Handler Testing Strategy

The route handlers are inline on `Router()` instances. To test them without supertest, there are two viable approaches:

### Approach A: Extract Handlers from Router Stack (Recommended)
Express Router exposes its middleware stack. You can find handlers by path and method:
```typescript
import { brandRouter } from '../../routes/brands';

// Router.stack contains Layer objects with route.path, route.methods, and route.stack
const getHandler = (router: any, method: string, path: string) => {
  const layer = router.stack.find((l: any) =>
    l.route?.path === path && l.route?.methods[method]
  );
  // layer.route.stack contains the middleware chain (auth + handler)
  // The last one is typically the actual handler
  return layer?.route?.stack[layer.route.stack.length - 1]?.handle;
};
```
**Downside:** Brittle if middleware order changes. But since auth is mocked as pass-through, this works.

### Approach B: Minimal Express App with supertest (Alternative -- NOT Recommended)
Would require installing supertest. Not recommended because the project convention uses direct handler invocation (see rbac.test.ts).

### Approach C: Direct Handler Invocation via Route Registration Spy (Best)
Mock the Router constructor so that `router.get()`, `router.post()`, etc. capture the handler arguments:
```typescript
// Instead of importing the router, intercept route registration
// This is the cleanest approach for isolated handler testing
```

**Final recommendation:** Use Approach A (extract from router stack) as the primary pattern. The route files export the router instance. Access `routerInstance.stack` to extract handlers. Mock `requireAuth` and `requireRole` in `../../middleware/auth` so they call `next()`. Then call the extracted handler with mock `req`/`res`/`next`.
