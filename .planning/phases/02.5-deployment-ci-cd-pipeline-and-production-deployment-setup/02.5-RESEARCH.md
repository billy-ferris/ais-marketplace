# Phase 02.5: Deployment — CI/CD Pipeline and Production Deployment Setup - Research

**Researched:** 2026-07-05
**Domain:** CI/CD (GitHub Actions), PaaS deployment (Railway + Vercel), production DB migrations (Drizzle), cross-origin auth/CORS (Clerk + Express)
**Confidence:** HIGH (verified against live repo + official docs; two production-blocking build landmines confirmed by direct execution)

## Summary

This phase stands up CI and a split-host production topology for a pnpm@9.15.0 + turbo monorepo: Express 4 API + PostgreSQL on Railway, Vite React SPA on Vercel. All locked decisions (D-01..D-11) are compatible with the current codebase **except one area the planner must resolve first**: the API's production start path (`node dist/index.js`, the current `start` script) is **completely broken today** and has never been run in production. I verified this by building and running it — two independent failures: (1) tsc emits extensionless relative imports that Node ESM refuses (`ERR_MODULE_NOT_FOUND` on `./routes/webhooks`), and (2) the `@ais/shared` workspace package exports raw `.ts` source with no build output, which Node cannot load from compiled JS. Dev works only because `tsx` tolerates both. **The lowest-friction, verified-working production start is `tsx src/index.ts`** (I booted it and `/api/health` returned `{"status":"ok"}`), which also preserves the monorepo's shared-TS-source pattern with zero build step. This requires promoting `tsx` and `drizzle-kit` from `devDependencies` to `dependencies` so they survive any production dependency pruning.

CI is the easy part: the 171-test API suite is fully mocked (`vi.mock`/`vi.hoisted`) and passes in ~2s **with an invalid `DATABASE_URL`** — I confirmed CI needs no Postgres service. The catch: D-03 specifies an **eslint** gate, but there is **no eslint config and no `lint` script anywhere** — `turbo run lint` currently executes 0 tasks and passes trivially. There is also no `typecheck` script for the required `tsc --noEmit`. Both are Wave 0 gaps.

Production migrations move from `db:push` to committed SQL via `drizzle-kit generate` + a Railway **pre-deploy command** running `drizzle-kit migrate` (non-interactive, runs in a separate container before the app serves). Cross-origin correctness needs an env-driven CORS allowlist (prod + Vercel preview pattern), Clerk **production instance** keys with `authorizedParties` set on `clerkMiddleware`, and Bearer-token auth flow (already how the SPA must call a different origin).

**Primary recommendation:** Run the API in production as `tsx src/index.ts` (promote `tsx`+`drizzle-kit` to `dependencies`); gate `main` with a GitHub Actions job running `turbo run typecheck lint test` (add eslint + typecheck scripts in Wave 0); apply migrations via a Railway `preDeployCommand`; drive CORS/Clerk from env allowlists covering the Vercel prod domain + preview pattern.

### Pattern 1: CI workflow (pnpm + turbo, no DB)
**What:** Single job runs the full gate; tests are mocked so no service containers are needed.
**When to use:** Every push to `main` + every PR (D-03/D-04).
```yaml
# Source: adapted from turborepo.dev/docs/guides/ci-vendors/github-actions
name: CI
on:
  push: { branches: [main] }
  pull_request: {}
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
jobs:
  verify:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4          # reads pnpm@9.15.0 from packageManager pin
      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: 'pnpm' }
      - run: pnpm install --frozen-lockfile
      - run: pnpm turbo run typecheck lint test
```
> Ordering matters: `pnpm/action-setup` MUST come before `setup-node` so `cache: 'pnpm'` can find the pnpm store. Do NOT set turbo remote-cache env vars unless a Vercel/turbo token is configured — omit `TURBO_TOKEN`/`TURBO_TEAM` for a local-cache-only CI.

### Pattern 2: Railway config-as-code (pre-deploy migration + tsx start)
**What:** Committed `railway.toml` builds with Nixpacks, runs `drizzle-kit migrate` in a pre-deploy container, then starts the API.
```toml
# Source: docs.railway.com/reference/config-as-code + deployments/pre-deploy-command
[build]
builder = "NIXPACKS"

[deploy]
# runs in a SEPARATE container with the app image + env (incl. DATABASE_URL) BEFORE serving
preDeployCommand = "pnpm --filter @ais/api exec drizzle-kit migrate"
startCommand = "pnpm --filter @ais/api exec tsx src/index.ts"
healthcheckPath = "/api/health"
restartPolicyType = "ON_FAILURE"
```
> `railway.json` is equivalent (`{ "$schema": "https://railway.com/railway.schema.json", "build": {...}, "deploy": {...} }`). If `preDeployCommand` fails (non-zero), the deploy aborts and does not roll forward — exactly the D-06 safety property. Railpack is Railway's newer default builder, but `builder = "NIXPACKS"` remains valid and satisfies D-10.

### Pattern 3: Env-driven CORS allowlist (replaces single-origin)
**What:** Replace the current single `FRONTEND_URL` origin with an allowlist that covers prod + Vercel preview URLs.
```typescript
// apps/api/src/index.ts — replaces the existing cors({ origin: FRONTEND_URL }) block
// Source: pattern from expressjs/cors README (origin function form)
const staticAllow = (process.env.CORS_ALLOWED_ORIGINS ?? 'http://localhost:5173')
  .split(',').map(s => s.trim()).filter(Boolean);
// Vercel preview URLs look like https://<project>-<hash>-<scope>.vercel.app
const previewRe = process.env.CORS_PREVIEW_PATTERN
  ? new RegExp(process.env.CORS_PREVIEW_PATTERN)
  : null;
app.use(cors({
  origin(origin, cb) {
    if (!origin) return cb(null, true);            // curl / same-origin / server-to-server
    if (staticAllow.includes(origin)) return cb(null, true);
    if (previewRe?.test(origin)) return cb(null, true);
    cb(new Error(`CORS: origin ${origin} not allowed`));
  },
  credentials: true,
}));
```
> **Keep `FRONTEND_URL` separate.** It is also used in `apps/api/src/services/email.ts` to build email CTA links — it must remain a single canonical prod URL, not a list. Introduce `CORS_ALLOWED_ORIGINS` (+ optional `CORS_PREVIEW_PATTERN`) as *new* vars for the allowlist; do not overload `FRONTEND_URL`.

### Pattern 4: Clerk cross-origin hardening
**What:** Set `authorizedParties` on `clerkMiddleware` to the same allowlist and use production keys.
```typescript
// Source: clerk.com/docs/reference/express/clerk-middleware
app.use(clerkMiddleware({
  authorizedParties: staticAllow,   // + previews if applicable; prevents subdomain-cookie/CSRF attacks
}));
```
Production also requires: `pk_live_*`/`sk_live_*` keys (not `pk_test`), Clerk Frontend-API DNS/CNAME setup for the production domain, and `allowed_origins` set to the Vercel prod domain + preview pattern (Clerk dashboard or PATCH the instance endpoint). Cross-origin means the SPA must send the session token as `Authorization: Bearer <token>` (via `getToken()`), which the existing `getAuth(req, { acceptsToken: 'session_token' })` already accepts.

### Pattern 5: Vercel SPA (monorepo)
**What:** Root Directory = `apps/web`, framework preset Vite, SPA rewrite.
```json
// vercel.json (repo root, or apps/web with Root Directory set)
{ "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }] }
```
Set Root Directory to `apps/web` in Vercel project settings; Vercel detects pnpm from the root `pnpm-lock.yaml` and the `pnpm@9.15.0` `packageManager` pin. Inject `VITE_API_BASE_URL` (→ Railway API URL) and `VITE_CLERK_PUBLISHABLE_KEY` as Vercel build-time env vars (D-08). PR previews are on by default (D-11).

### Anti-Patterns to Avoid
- **Applying migrations in the app start path or on request handlers** — D-06 forbids it; use the pre-deploy container.
- **Running `db:push` against production** — destructive/interactive; dev-only (D-05).
- **Running `pnpm db:seed` in prod** — `assertNotProduction()` throws when `NODE_ENV=production`; the pre-deploy step must run **migrate only**, never seed.
- **Overloading `FRONTEND_URL` as the CORS list** — it feeds email links; keep it a single URL.
- **Setting `NODE_TLS_REJECT_UNAUTHORIZED=0` in prod** — dev-only cert workaround; real certs verify fine.
- **Hardcoding a pnpm version in `pnpm/action-setup`** — let v4 read the `packageManager` pin to avoid drift.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Apply pending migrations at deploy | A custom "check schema + run SQL" script | `drizzle-kit migrate` in `preDeployCommand` | Tracks applied migrations in a journal table, non-interactive, idempotent [CITED: orm.drizzle.team/docs/migrations] |
| Run migration before serving | App-startup migration code + locking | Railway `preDeployCommand` (separate container, blocks deploy on failure) | Built-in ordering + failure-aborts-deploy semantics [CITED: docs.railway.com/deployments/pre-deploy-command] |
| SPA deep-link 404 handling | Custom server / catch-all route | `vercel.json` rewrite all→`/index.html` | One-line platform feature |
| pnpm store caching in CI | Manual cache keys | `actions/setup-node` `cache: 'pnpm'` | Keys off lockfile automatically |
| Cross-origin token verification | Manual JWT/JWKS verification | `@clerk/express` `clerkMiddleware` + `authorizedParties` | Already wired; just add allowlist |

**Key insight:** Every hard part of this phase (ordered migration, preview URLs, SPA routing, JWT verification) is a first-class platform/library feature. The only genuinely custom code is the CORS allowlist function and the eslint config.

## Common Pitfalls

### Pitfall 1: `node dist/index.js` production start is broken (CONFIRMED)
**What goes wrong:** The current `start` script (`node dist/index.js`, invoked by D-10's `pnpm --filter @ais/api start`) crashes on boot in production.
**Why it happens:** Two independent, verified failures — (1) `tsc` (with `moduleResolution: bundler`) emits **extensionless** relative imports (`import ... from './routes/webhooks'`), which Node ESM rejects → `ERR_MODULE_NOT_FOUND`; (2) `@ais/shared` `exports` point at raw `.ts` files (`./src/constants/index.ts`) with no JS build, which Node cannot load. Dev survives only because `tsx` transparently handles both.
**How to avoid:** Set the Railway start command to `tsx src/index.ts` (verified: boots and `/api/health` → `{"status":"ok"}`), and **promote `tsx` + `drizzle-kit` from `devDependencies` to `dependencies`** in `apps/api` so they exist in the production image. Either point the `start` script at tsx, or set `startCommand` explicitly in `railway.toml`.
**Warning signs:** `ERR_MODULE_NOT_FOUND` / `ERR_UNKNOWN_FILE_EXTENSION` in Railway deploy logs; container restart loop.

### Pitfall 2: `NODE_ENV=production` prunes the tools you need
**What goes wrong:** `drizzle-kit migrate` (pre-deploy) or `tsx` (start) is "not found" in the Railway image.
**Why it happens:** With `NODE_ENV=production` set as a service variable (required by D-08 for the seed guard), pnpm/Nixpacks may skip/prune `devDependencies` — and today both `tsx` and `drizzle-kit` are devDeps. [CITED: pnpm.io/cli/install — `--prod` skips devDependencies]
**How to avoid:** Promote `tsx` and `drizzle-kit` to `dependencies` (Pitfall 1 fix already does this). Then dependency pruning is irrelevant.
**Warning signs:** `drizzle-kit: not found` / `tsx: not found` in pre-deploy or start logs.

### Pitfall 3: eslint gate is vaporware (CONFIRMED)
**What goes wrong:** CI is "green" but nothing was linted — D-03's eslint requirement is silently unmet.
**Why it happens:** No eslint config and no `lint` script exist anywhere; `turbo run lint` reports "No tasks were executed" and exits 0.
**How to avoid:** Wave 0 — add a flat eslint config (`eslint.config.js`) + `lint` scripts in each package (or a root lint), and a `typecheck` script (`tsc --noEmit`) + turbo `typecheck` task. Then CI runs `turbo run typecheck lint test`.
**Warning signs:** CI logs show `lint` running 0 packages; PRs with lint errors merge clean.

### Pitfall 4: Drizzle baseline vs. existing `db:push` drift
**What goes wrong:** `drizzle-kit generate` produces a baseline migration, but if pointed at a DB previously shaped by `db:push`, `migrate` may conflict.
**Why it happens:** `db:push` mutates schema without recording migrations; the journal starts empty.
**How to avoid:** Production Postgres is **brand new/empty**, so the baseline `0000_*.sql` applies cleanly. Generate the baseline from current schema, commit it, and never run `db:push` against prod. (Dev DBs that were push-shaped are irrelevant to prod.)
**Warning signs:** `migrate` errors like "relation already exists" — indicates it ran against a push-shaped DB.

### Pitfall 5: Clerk still on test instance in prod
**What goes wrong:** 401s / auth failures from the Vercel origin.
**Why it happens:** Using `pk_test`/`sk_test` keys, missing DNS/CNAME for the production Frontend API, or Vercel origin not in Clerk `allowed_origins`/`authorizedParties`.
**How to avoid:** Provision the Clerk **production instance**, complete DNS, set `allowed_origins` (prod + preview) and `authorizedParties` on `clerkMiddleware`. Send tokens as `Authorization: Bearer`.
**Warning signs:** `getAuth` returns `userId: null` from prod; CORS-looking failures that are actually auth-origin rejections.

### Pitfall 6: Vercel uses npm despite pnpm workspace
**What goes wrong:** Build fails to resolve `@ais/shared` or uses the wrong lockfile.
**Why it happens:** Package-manager mis-detection when Root Directory is `apps/web` and install runs there instead of repo root. [CITED: vercel.com community + monorepo docs]
**How to avoid:** Rely on the root `pnpm-lock.yaml` + `packageManager: pnpm@9.15.0` pin; if needed, set install command to run at the workspace root. Verify the build log shows pnpm, not npm.
**Warning signs:** `npm` in Vercel build logs; "Cannot find module @ais/shared".
