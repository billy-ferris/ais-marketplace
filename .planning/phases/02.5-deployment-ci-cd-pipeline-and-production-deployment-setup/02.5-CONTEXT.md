# Phase 02.5: Deployment — CI/CD Pipeline and Production Deployment Setup - Context

**Gathered:** 2026-07-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Stand up automated CI and a production deployment path for the AIS Marketplace monorepo (Express API + Vite React SPA + PostgreSQL). Delivers: a GitHub Actions CI pipeline that gates `main`, a hosted production topology, a safe production database-migration flow, and per-environment secret/config management — so the app can ship to a live environment.

**In scope:** CI workflow, hosting/deploy config for API + web + DB, production migration strategy, environment/secrets wiring, cross-origin (CORS/Clerk) config for the split frontend/backend hosts.

**Out of scope (own phases / already done):** application features, the storefront (Phase 3), monitoring/observability tooling and autoscaling tuning (future), and the security hardening already completed in Phase 02.4.
</domain>

<decisions>
## Implementation Decisions

### Hosting Topology
- **D-01:** API + PostgreSQL on **Railway**; web SPA on **Vercel**. Split frontend/backend hosts (two providers, two dashboards/secret stores).
- **D-02:** This is a **cross-origin** deployment — the Vercel-hosted SPA calls the Railway-hosted API on a different origin. CORS on the API and Clerk allowed-origins must both account for this (see D-08).

### CI Pipeline (GitHub Actions)
- **D-03:** CI runs on push + pull_request: **eslint + `tsc --noEmit` typecheck + the full API test suite** (171 tests, ~2s). Use pnpm with the repo's `pnpm@9.15.0`; leverage turbo tasks (`lint`, `build`, `test`).
- **D-04:** **Branch protection gates `main`** — CI must be green to merge. No CI exists yet; this is greenfield.

### Production Database Migrations
- **D-05:** Move production off drizzle **`db:push`** to **generated SQL migrations**: dev edits schema → `drizzle-kit generate` → commit the SQL → migrations are reviewable in PRs. `db:push` remains a dev-only convenience.
- **D-06:** Migrations apply via a **Railway release / pre-deploy step** (`drizzle-kit migrate`) that runs **before** the API process starts serving — never serve against an un-migrated DB, never apply on hot request paths.

### Secrets & Environment Config
- **D-07:** Real secrets live in **host-native env stores** (Railway env for API/DB, Vercel env for web), managed **per environment**. The committed **`.env.example`** files are the source-of-truth contract for required keys (already exist for both apps).
- **D-08:** Production correctness rules: set **`NODE_ENV=production`** (activates the Phase 02.4 seed guard `assertNotProduction`); the dev-only **`NODE_TLS_REJECT_UNAUTHORIZED=0` must NOT be set in prod**. Web **`VITE_*`** vars (incl. `VITE_API_BASE_URL` → Railway API URL, `VITE_CLERK_PUBLISHABLE_KEY`) are injected at **Vercel build time**.
- **D-09:** CORS on the API is an **env-var-driven allowlist** covering the Vercel production domain **and** the Vercel preview-URL pattern. The same origins are allow-listed in **Clerk** (production instance).

### Build & Runtime
- **D-10:** Railway builds the API with **Nixpacks** (zero-config auto-detect for the pnpm monorepo) with an explicit start command `pnpm --filter @ais/api start`. No committed Dockerfile unless Nixpacks proves insufficient.
- **D-11:** Vercel serves the web build (`apps/web` → `dist`) with an **SPA rewrite** (all routes → `index.html` for client-side routing) and **PR preview deploys enabled** alongside the GitHub Actions CI gate.

### Claude's Discretion
- Exact GitHub Actions YAML structure (job matrix, caching of pnpm store/turbo, concurrency), Railway service/config file format (`railway.toml` vs dashboard), Vercel project config (`vercel.json` rewrites), and the precise env-var names for the CORS allowlist — planner/researcher to choose per current provider best practices.
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **turbo tasks** (`build`, `lint`, `test`): CI can invoke these directly across the workspace.
- **Per-app `start`/`build` scripts**: `apps/api` has a `start` script (prod entry); `apps/web` has `build`→`dist` + `preview`.
- **`.env.example` files**: already present for both apps — the documented key contract for host env stores.
- **171 passing API tests**: the ready-made CI quality gate (fast, mocked-db harness).

### Established Patterns
- **Monorepo:** pnpm@9.15.0 workspaces + turbo; tasks run from package dirs.
- **Auth:** custom `requireAuth()` in `apps/api/src/middleware/auth.ts` (returns 401 JSON for SPA fetches) — cross-origin deploy must keep Clerk token flow working from the Vercel origin.
- **DB:** Drizzle ORM; dev uses `db:push`; `db:generate` available for the new migration flow.

### Integration Points
- **CORS config in the API** (Express) must be extended to an env-driven allowlist for the Vercel origin(s) — locate and wire the existing CORS setup.
- **`VITE_API_BASE_URL`** (apps/web) must point at the Railway API URL per environment; injected at Vercel build.
- **Clerk** production instance: allowed origins must include the Vercel prod domain + preview pattern.
- **Railway release step** hooks the migrate command into the deploy lifecycle before API boot.
</code_context>

<specifics>
## Specific Ideas

- Split-host topology is deliberate: Vercel for the SPA (CDN + automatic PR previews), Railway for API + Postgres (single backend provider, matches the existing PROJECT.md decision).
- Safety emphasis carried from Phase 02.4: production must never run destructive `db:push`, must set `NODE_ENV=production`, and must exclude the dev-only TLS-reject flag.
</specifics>
