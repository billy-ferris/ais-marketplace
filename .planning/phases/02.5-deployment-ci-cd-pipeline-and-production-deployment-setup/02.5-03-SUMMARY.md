## Deferred Human Provisioning Steps

The following steps require a live GitHub remote and repository admin access. They were deliberately NOT executed (no git remote is configured in this repo; provisioning is a human responsibility). Execute them once the repo is pushed to GitHub.

### 1. Push the branch / commit so GitHub registers the workflow

The `verify` check name must have run at least once before it becomes selectable as a required status check.

```bash
# (Prerequisite) configure a remote if one does not yet exist:
git remote add origin git@github.com:<org>/<repo>.git

# Push main (or open a PR from a feature branch) so the CI workflow runs:
git push -u origin main
```

### 2. Open a PR so the check runs at least once

Open a pull request (any branch → `main`). Confirm the **CI / verify** job appears and completes in the PR's Checks tab. This registers the `verify` status-check name for selection in the branch protection UI.

### 3. Enable branch protection on `main` (GitHub UI) — D-04

GitHub → repo → **Settings → Branches → Branch protection rules → Add rule** for `main`:

1. Set **Branch name pattern**: `main`.
2. Enable **Require a pull request before merging**.
3. Enable **Require status checks to pass before merging**.
4. In the status-checks search box, select the CI job **`verify`** as a required check.
5. Enable **Require branches to be up to date before merging**.
6. Click **Save changes**.

### 4. Confirm the gate is live

Open a PR containing a deliberate lint/type/test failure and confirm the merge button is blocked until CI is green. This proves the T-025-03-T mitigation (branch protection makes the CI check a required gate) is active.
